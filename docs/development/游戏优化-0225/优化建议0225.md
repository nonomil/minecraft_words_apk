## 基于真实代码的优化建议

### 一、性能问题（当前最值得修）

**`findSupportingPlatform` 每帧全量遍历**

```js
// 当前：每帧对所有平台做线性扫描
for (let i = 0; i < platforms.length; i++) { ... }
```

这个函数被 `updateEntityGravity` 调用，而 `updateEntityGravity` 被 items、chests 每帧调用。平台数量一多，开销是 O(n × entities)。建议按 X 坐标分桶（spatial hash），只检测附近的平台：

```js
// 改进：只检查 X 范围在 ±200px 内的平台
const nearPlatforms = platforms.filter(p => 
    !p.remove && 
    p.x < entityX + entityWidth + 200 && 
    p.x + p.width > entityX - 200
);
```

**`decorations.filter` + `enemies.filter` 每帧重建数组**

```js
decorations = decorations.filter(d => d.x + d.width > cameraX - removeThreshold && !d.remove);
enemies = enemies.filter(e => !e.remove && e.y < 950);
```

每帧都在重建整个数组，GC 压力大。移动端低端机上会造成帧率抖动。建议改为标记删除 + 定期清理（每 60 帧清一次）：

```js
// 每60帧才真正清理一次
if (gameFrame % 60 === 0) {
    decorations = decorations.filter(d => !d.remove && d.x + d.width > cameraX - removeThreshold);
    enemies = enemies.filter(e => !e.remove);
}
```

### 二、架构问题（全局变量泛滥）

`13-game-loop.js` 里直接用了 `platforms`、`enemies`、`items`、`chests`、`golems`、`projectiles`、`decorations`、`particles`、`wordGates`、`floatingTexts`、`inventory`、`player`、`score`、`gameFrame`、`cameraX`... 大量全局变量，模块间没有边界。

这不是立刻要重构的问题，但会带来一个具体 bug 风险：**`paused` 和 `pausedByModal` 状态容易失步**。

当前逻辑：
```js
function showInventoryModal() {
    pausedByModal = !paused;  // 如果已经 paused，就不记录
    paused = true;
}
function hideInventoryModal() {
    if (pausedByModal) paused = false;
    pausedByModal = false;
}
```

问题：如果用户在 BOSS 战暂停状态下打开背包，再关闭背包，游戏会意外恢复（`paused` 被置为 false）。建议改成计数器或栈：

```js
let pauseStack = 0;
function pushPause() { pauseStack++; paused = true; }
function popPause()  { pauseStack = Math.max(0, pauseStack - 1); if (pauseStack === 0) paused = false; }
```

### 三、`damagePlayer` 逻辑复杂度过高

这个函数一口气做了：击退、位置合法性校验、伤害计算、难度缩放、护甲减伤、护甲耐久消耗、积分惩罚、UI更新、游戏结束判断——230行左右的超级函数。

直接后果是：护甲减伤逻辑里有一个潜在 bug：

```js
const reductionByArmor = {
    leather: 0.2, chainmail: 0.35, iron: 0.55,
    gold: 0.65, diamond: 1.0, netherite: 1.0
};
const baseReduction = Math.min(0.6, defense * 0.12);
const armorReduction = reductionByArmor[armorId];
const reduction = hasArmor ? (Number.isFinite(armorReduction) ? armorReduction : baseReduction) : baseReduction;
const minDamage = hasArmor ? 0 : 1;
const actualDamage = Math.max(minDamage, Math.round(scaledDamage * (1 - reduction)));
```

**diamond 和 netherite 的 reduction 是 1.0，即 `1 - 1.0 = 0`，但 `minDamage = 0`**，所以穿钻石/下界合金理论上完全无敌。这可能是有意设计，但逻辑没有任何注释，维护时很容易被误改。

建议最低限度：把计算逻辑提取成 `calcActualDamage(baseDamage, armorId, durability)` 纯函数，方便单独测试。

### 四、`useInventoryItem` 超长 if-else 链

这个函数是一个巨大的 if-else，每种物品硬编码一个分支，现在已经有约15种物品。每次新增物品都要改这个函数，而且物品数据（`FOOD_TYPES`、`ITEM_LABELS`）分散在其他模块里，耦合严重。

建议把每种物品的使用逻辑注册到一个 map：

```js
const ITEM_HANDLERS = {
    gunpowder: (inv, player) => {
        inv.gunpowder--;
        bombs.push(new Bomb(player.x, player.y, player.facingRight ? 1 : -1));
        showToast("💣 投掷炸弹");
    },
    diamond: (inv, player) => {
        if (playerHp >= playerMaxHp) { showToast("❤️ 已满血"); return false; }
        inv.diamond--;
        healPlayer(1);
        showToast("💎 恢复生命");
    },
    // ...
};

function useInventoryItem(itemKey) {
    const handler = ITEM_HANDLERS[itemKey];
    if (handler) {
        const result = handler(inventory, player);
        if (result !== false) { updateHpUI(); updateInventoryUI(); }
    }
}
```

这样新增物品只需在 `ITEM_HANDLERS` 加一条，不需要动主函数。

### 五、HTML 结构问题（`Game.html`）

**25个 `<script>` 标签串行加载**


建议在 `scripts/sync-web.js` 合并时加一个简单的全局变量冲突检测：

```js
// 合并前先扫描每个模块的顶层 var/let/const/function 声明
// 输出重复名称的警告
```

**`onclick="window.useInventoryItem && window.useInventoryItem('${entry.key}')"` 模式**

`renderInventoryModal` 里大量使用了 innerHTML 字符串拼接 + 内联 onclick：

`entry.id` 如果来自用户存档（localStorage），这里存在 XSS 风险——虽然是本地游戏、用户自己的数据，但习惯上还是应该用 `addEventListener` 代替内联 onclick，这样也便于调试和测试。

### 六、卡住检测逻辑有误

```js
if (hasInput && Math.abs(player.x - player._lastStuckX) < 0.5 && player.grounded) {
    player._stuckFrames++;
    if (player._stuckFrames > 45) {
        player.y = player.y - blockSize * 0.8;  // 强制向上推
        player.velY = -2;
        player._stuckFrames = 0;
    }
} else {
    player._stuckFrames = 0;  // ← 问题在这里
}
player._lastStuckX = player.x;  // ← 每帧都更新了
```

**Bug**：`_lastStuckX` 在判断之后立即更新，所以下一帧的比较 `player.x - player._lastStuckX` 永远是当帧位移，而不是累积位移。卡住45帧的条件实际上永远很难触发——每帧都重新记录了起点。

修复：只在 `_stuckFrames` 递增时不更新 `_lastStuckX`：

```js
if (hasInput && player.grounded) {
    if (Math.abs(player.x - player._lastStuckX) < 0.5) {
        player._stuckFrames++;
        // 不更新 _lastStuckX，保持原始卡住位置
    } else {
        player._stuckFrames = 0;
        player._lastStuckX = player.x;  // 正常移动时才更新
    }
    if (player._stuckFrames > 45) {
        player.y -= blockSize * 0.8;
        player.velY = -2;
        player._stuckFrames = 0;
        player._lastStuckX = player.x;
    }
} else {
    player._stuckFrames = 0;
    player._lastStuckX = player.x;
}
```

## 游戏性问题

### 1. 武器切换体验断层

从 `Game.html` 的 HUD 可以看出，武器栏只在底部的 `inventory-bar` 显示数量，没有「当前武器」的视觉高亮（除了 `equip-status` 文字）。玩家在手机上很难感知自己在用什么武器。

更大的问题在 `handleAttack` 里：

```js
if (weapon.type === "ranged") {
    if (mode === "tap") {
        releaseBowShot(0.35);  // tap直接发射
        return;
    }
    if (!playerWeapons.isCharging) {
        startBowCharge();      // 长按才蓄力
    }
    return;
}
```

弓的「tap = 弱射，长按 = 蓄力射」对触屏用户非常不直观，而且 `tap` 和 `长按` 的判断在 `16-events.js` 里处理，容易出现误触——手指还没抬起就触发了 tap，蓄力逻辑完全没机会启动。

### 2. 镐的挖掘反馈缺失

`game-loop.js` 里调用了 `triggerGravityCheck(digLeft, digRight, platformY)` 让方块坍塌，但从代码结构看，挖掘动作本身（`digGroundBlock`）在 `04-weapons.js` 里，没有明显的「挖掘中」动画状态。玩家不知道镐在起作用。

### 3. Boss 战节奏单调的根本原因

`bossArena.checkSpawn()` 和 `bossArena.update()` 每帧都在 `update()` 主循环里调用，但 Boss 战期间的「武器锁定为剑」（`isBossWeaponLockActive`）和「视口锁定」是硬切换，缺乏过渡动画。玩家从跑酷模式突然进入 Boss 战，没有任何仪式感铺垫（如黑屏、Boss 登场动画）。

### 4. 水下物理和陆地物理割裂

```js
if (isUnderwater) {
    player.velX *= 0.9;  // 水下摩擦
} else {
    player.velX *= gameConfig.physics.friction;  // 陆地摩擦
}
```

两套物理参数完全独立，进出水面时速度会突然跳变。特别是从陆地落入水中时，没有「入水」冲击减速，感觉像穿模。

## 学习性问题（基于 HTML 结构深度推断）

### 5. 挑战题触发时机破坏心流

从 `maybeTriggerLearningChallenge(item.wordObj)` 的调用位置来看，挑战是在「捡到单词道具的瞬间」触发的：

```js
// item.collected = true → speakWord → maybeTriggerLearningChallenge
```

这意味着玩家刚捡到单词、还没完整听完朗读，挑战弹窗就可能出现。孩子的反应是：**单词刚听一遍就要考**，记忆负担极大，容易挫败。

建议改为：捡到单词只触发单词卡展示，挑战在「下一个单词道具生成时」或「5秒后」触发，给孩子一个短暂的记忆窗口。

### 6. 单词卡停留时间太短，学习效果存疑

设置里最长只有 1.5 秒：
```html
<option value="1500">更长(1.5秒)</option>
```

英语单词的有效记忆需要至少看清单词 + 听发音 + 默念一遍，1.5秒对拼音还没完全掌握的小学低年级学生完全不够。建议把上限提升到 3-4 秒，并在单词卡上加「下一个」按钮让孩子自己控制节奏。

### 7. 答错惩罚设计对孩子不友好

从 README 和设置可知：答错默认扣 8 分。对于**正在学习阶段的孩子**来说，答错是正常的，扣分会让孩子：
- 害怕答错 → 不敢猜 → 跳过挑战
- 积分下降 → 关卡倒退 → 挫败感

建议把答错的惩罚设计改为**「答错不扣分，但不给奖励」**，同时在答错后立刻展示正确答案和朗读，形成纠错记忆。扣分模式可以作为「挑战模式」单独开关。

### 8. 词语连线复活的 UI 体验问题

从 `Game.html` 的 word-match-screen 结构：

```html
<div class="match-container">
    <div class="match-column" id="match-left"></div>
    <svg class="match-lines" id="match-lines"></svg>
    <div class="match-column" id="match-right"></div>
</div>
```

5组单词 × 2列，用 SVG 连线，在手机竖屏上空间极其拥挤。5组单词要在同一屏显示，每个单词区域只有大约 40-50px 高度，手指很难精确点击。这是个已知的「在手机上几乎玩不了」的设计。

建议：连线改为「卡片配对翻牌」或「下拉选择」，触屏友好得多。

### 9. 「短语跟随」逻辑对低年级学生过于复杂

高级设置里有 `phraseFollowMode`（混合/紧跟/隔词/关闭），这是个对**成人学习者**有价值的设计，但对 6-10 岁孩子来说，家长根本不知道怎么配置。

更关键的是：「隔词出现短语」打断了游戏节奏，孩子正在打怪，突然弹出一个关联上一个单词的短语，时间上下文已经断了，反而造成混乱。

建议：把短语跟随默认关闭，只在「单词学习屋」（村庄内）展示短语和例句，游戏主流程只展示单词本身。

### 10. 学习数据没有跨局持久化利用

从 `getLearnedWordCount()` 的实现：
```js
function getLearnedWordCount() {
    const vocab = progress && progress.vocab ? Object.keys(progress.vocab) : [];
    return vocab.length;
}
```

只统计了「见过的词数」，但没有区分「见过」和「真正掌握」。更重要的是：**这个数据完全没有反馈给玩家**——孩子和家长都看不到「这周学了哪些词」。档案页（`profile-modal`）只显示：游戏次数、最高积分、已学词数、游玩时长，没有「单词本」入口。
