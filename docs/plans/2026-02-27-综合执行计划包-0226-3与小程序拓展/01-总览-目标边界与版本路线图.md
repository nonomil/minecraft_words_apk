# 总览：目标边界与版本路线图（执行版）

## 1. 目标拆解（90天）

1. 经营目标：先稳定小额成交，再考虑扩展。
2. 产品目标：先减少售后，再增强学习感知。
3. 技术目标：先稳 APK/Web，再谈小程序与云能力。

## 2. 版本顺序（禁止跳跃）

1. vA（P0，1-2天）：运营前修正 + 3项最小开发。
2. vB（P1，2-4天）：学习数据与反馈体验增强。
3. vC（P1，2-3天）：Web TTS 可诊断 + PWA 可控更新。
4. vD（P2，3-5天）：统一 TTS Provider，准备小程序迁移。
5. vE（P3，可选，4-7天）：班级房间码 + 云存档 MVP。

## 3. 资源预算

1. 每天投入：60-120 分钟。
2. 每周可执行容量：6-10 小时。
3. 同时进行任务上限：2 个（1 个代码 + 1 个内容运营）。

## 4. 依赖图

1. vA -> vB（先把成交风险压下来）。
2. vB -> vC（有数据可展示再做网页演示闭环）。
3. vC -> vD（先稳定 Web 语音，再抽象多端 Provider）。
4. vD -> vE（有跨端基础再谈班级云能力）。

## 5. 通用验收标准

1. 功能验收：页面上看得见，点击能触发，异常有提示。
2. 测试验收：关键脚本可跑，至少手工回归 + 核心 E2E。
3. 运营验收：退款率、咨询转化率、售后耗时三指标有记录。

## 6. 通用回滚标准

满足任一条，回滚当前版本至上一个稳定点：

1. 核心流程不可用（无法进游戏/无法保存/无法发音）。
2. 新增功能引发明显售后增量（连续3天）。
3. 修复成本超过预估 2 倍且无明显收益。
