# 风险边界与止损线（轻量路线）

## 1. 当前路线边界

1. 不做应用商店上架
2. 不做重资质投入
3. 不做公众号长期运营
4. 不做高成本投放

以上边界确保项目保持“业余可持续”。

## 2. 主要风险

1. 渠道波动风险（小红书流量不稳定）
2. 交付风险（用户设备差异导致安装问题）
3. 内容疲劳（素材重复导致转化下降）
4. 售后挤占时间（影响持续输出）

## 3. 风险应对

1. 渠道波动
- 维持“双渠道”：小红书引流 + 咸鱼成交

2. 交付风险
- 安装教程标准化
- 常见问题模板化

3. 内容疲劳
- 固定三类选题轮换（机制/反馈/答疑）

4. 时间挤占
- 每日固定时长上限（90分钟）
- 超时不加班，避免长期断更

## 4. 止损线（满足任意两条即收缩）

1. 连续4周成交 < 10 单
2. 退款率连续4周 > 15%
3. 每周售后耗时 > 6 小时
4. 内容发布无法稳定达到每周 2 条

## 5. 收缩策略（止损后执行）

1. 只保留1个主商品（全量包）
2. 降低发帖频率到每周2条
3. 暂停新增功能开发，只做稳定性修复
4. 只维持最省时渠道动作

## 6. 继续投入条件（反向信号）

1. 连续8周每周成交 >= 15 单
2. 退款率稳定 < 8%
3. 转介绍占比 > 20%

满足后再考虑：

1. Web试玩页强化
2. 小程序迁移验证
3. 更系统化的品牌包装

## 审查意见（2026-02-27）

### 风险识别完整性

文档列出的四类风险（渠道波动、交付风险、内容疲劳、售后挤占时间）覆盖了主要场景，判断准确。

**需要补充的风险：设备兼容性**

- 当前未列出，但这是最可能导致退款的技术风险。
- 具体表现：Android 7 以下机型 Web Speech API 缺失，TTS 完全不可用；部分国产 ROM 的 WebView 版本过旧，可能导致游戏渲染异常。
- 应对：在咸鱼商品详情中明确标注"建议 Android 8.0 及以上系统"，提前过滤不兼容用户，减少售后和退款。

### 止损线评估

- **连续4周成交 < 10 单**：合理，4周是足够的验证窗口。
- **退款率连续4周 > 15%**：合理，但建议同时追踪退款原因，区分"产品问题"和"安装问题"——后者可通过 SOP 优化解决，不应触发整体收缩。
- **每周售后耗时 > 6 小时**：这是最容易被忽视的止损线，也是最重要的。建议在第1周就开始计时，不要等到超标才意识到。

### 继续投入条件评估

- "连续8周每周成交 >= 15 单"是合理的门槛，但 8 周周期较长。建议增加一个**早期信号**：如果前4周累计成交 >= 30 单且退款率 < 10%，可以提前启动 Web 试玩页建设，不必等满 8 周。
- Web 试玩页（直接部署 `Game.html`）技术成本极低，可以作为"继续投入"的第一步，而非等到所有条件都满足后才做。

### 边界声明与 IP 风险的关联

- 文档第1条"不做应用商店上架"直接规避了 IP 风险的最高曝光场景，判断正确。
- 补充：即使在咸鱼/小红书私域路线下，如果单月成交超过一定规模（参考：月流水超过 5000 元），IP 风险敞口会上升，建议在"继续投入条件"触发时同步启动换皮评估，而不是等到正式上架阶段才处理。
