# 第一阶段奖励闭环实施计划

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 在不推翻现有跑跳战斗框架的前提下，把“学习 → 奖励 → 继续玩”串成一个更强的正反馈闭环，并优先服务小学低年级用户。

**Architecture:** 方案以“少量新增配置 + 复用现有学习/宝箱/UI/掉落模块”为核心，不新增重型系统，优先在已有的挑战、收词、宝箱和 HUD 链路上加奖励钩子。第一阶段只做短链路奖励、轻量连胜和低年级减压，不同步追求幼儿园重设计或初中深度模式。

**Tech Stack:** HTML5、CSS3、JavaScript (ESM)、Canvas UI、现有配置加载机制、现有 e2e 测试框架（如适用）

---

## 0. 计划前提

### 已确认的产品结论

1. 第一阶段目标是“最快可落地的改进版”，不是重做产品。
2. 第一阶段主目标用户锁定为“小学低年级（1–3 年级）”。
3. 第一阶段不大改跑跳战斗框架。
4. 第一优先先做“奖励闭环”，第二优先再补“降低挫败”，第三优先补“场景记忆强化”。

### 本计划边界

**本计划包含：**
- 学习表现影响即时奖励
- 学习表现影响宝箱/掉落质量
- 轻量学习连胜与短时增益
- 小学低年级阶段的轻度减压调优
- UI 上的即时反馈与结果可视化

**本计划不包含：**
- 幼儿园模式重设计
- 初中深度模式
- 群系大扩展
- 敌人系统大改
- 新的大型账号/云存档功能

---

## 1. 目标体验

玩家在第一阶段应明显感受到以下变化：

1. 答对或学会单词后，马上拿到“看得见、用得上”的奖励。
2. 学习表现越好，宝箱、掉落和短时战斗体验越好。
3. 前期挫败感下降，但挑战感仍然存在。
4. 小学低年级用户更容易理解“学会 = 变强 = 更想继续玩”。

### Task 1：奖励配置骨架与全局状态

**Files:**
- Create: `config/reward-loops.json`
- Modify: `src/modules/01-config.js`
- Modify: `src/modules/17-bootstrap.js`
- Modify: `docs/plan/游戏优化设计/2026-03-06-游戏性学习性趣味性优化分析与第一阶段建议.md`

**目标：**
建立奖励闭环的配置入口，避免后续把阈值、倍率、持续时间写死在多个模块里。

**Step 1: 新建奖励配置文件**

在 `config/reward-loops.json` 中定义：
- `challengeRewards`
- `collectionRewards`
- `streakRewards`
- `chestBonusRules`
- `elementaryLowerTuning`

至少包含以下字段：
- 连续答对阈值
- 增益持续时长
- 宝箱品质加权修正
- 低年级答题时长
- 低年级扣分/惩罚权重

**Step 2: 在 `01-config.js` 增加全局状态**

新增或扩展以下命名状态：
- `rewardLoopConfig`
- `learningRewardState`
- `learningStreakState`
- `recentLearningPerformance`

要求：
- 状态结构清晰，避免魔数
- 命名与现有配置风格一致
- 为后续 UI、宝箱、挑战逻辑提供统一读入口

**Step 3: 在 `17-bootstrap.js` 接入配置加载**

实现启动阶段的配置加载与兜底逻辑：
- 成功加载 `config/reward-loops.json`
- 加载失败时回退到内置默认值
- 避免阻塞已有游戏启动流程

**Step 4: 校验配置接入**

手工验证：
- 游戏启动不报错
- 控制台能看到奖励配置已加载
- 不开启新逻辑时，原有流程仍可运行

### Task 2：挑战答对后的即时奖励链路

**Files:**
- Modify: `src/modules/12-challenges.js`
- Modify: `src/modules/13-game-loop.js`
- Modify: `src/modules/10-ui.js`
- Modify: `Game.html`

**目标：**
把“答对题目”从纯分数反馈改成“即时可感知奖励”。

**Step 1: 扩展挑战结算数据**

在 `12-challenges.js` 的正确/错误结算中输出统一结果对象，至少包含：
- `isCorrect`
- `rewardType`
- `rewardValue`
- `streakDelta`
- `difficultyProfile`

要求：
- 不破坏现有得分和钻石逻辑
- 新结果对象可被主循环或 UI 复用

**Step 2: 在主循环接入即时奖励**

在 `13-game-loop.js` 接入奖励应用函数，例如：
- `applyLearningReward(result)`
- `grantTemporaryBuff(type, duration)`
- `updateLearningStreak(result)`

第一批推荐奖励：
- 短时护盾
- 短时移速提升
- 额外宝石/材料
- 下一次宝箱品质加成标记

**Step 3: 在 UI 中可视化结果**

在 `10-ui.js` 和 `Game.html` 增加轻量提示：
- “答对 + 护盾”
- “连胜 3 次 + 掉落加成”
- “学习奖励已激活”

要求：
- 反馈要即时
- 文案尽量儿童可理解
- 不遮挡主画面核心操作区

**Step 4: 手工验收**

检查以下场景：
- 答对后立刻出现奖励提示
- 奖励效果在角色或 HUD 上可感知
- 连续答题时状态可持续更新
- 原有挑战流程未中断

**Step 5: Commit**

```bash

git commit -m "feat: add immediate rewards for learning challenges"
```

### Task 3：把宝箱和学习表现绑定

**Files:**
- Modify: `src/modules/15-entities-base.js`
- Modify: `src/modules/13-game-loop.js`
- Modify: `src/modules/10-ui.js`

**目标：**
让玩家感受到“最近学得好，接下来奖励就更好”。

**Step 1: 增加最近学习表现快照**

在全局或主循环中维护最近若干次学习行为的表现数据，至少记录：
- 最近正确率
- 当前连胜数
- 最近是否触发学习奖励
- 宝箱品质加成是否待消费

**Step 2: 修改宝箱品质或掉落权重**

在 `15-entities-base.js` 的宝箱或掉落逻辑中，接入学习表现加权：
- 连胜时提高 rare / epic 权重
- 最近挑战表现好时追加材料/钻石奖励
- 消费一次后清除“下一箱奖励加成”标记

要求：
- 加成必须可控，不能破坏整体经济
- 高品质奖励必须仍保留随机性
- 不要把普通玩家完全排除在奖励之外

**Step 3: 在 UI 中提示来源**

当宝箱收益因学习表现提高时，显示轻量解释：
- “学习表现加成生效”
- “连胜奖励：宝箱升级”

让玩家理解因果关系，而不是误以为纯随机。

**Step 4: 手工验收**

验证：
- 表现好时，宝箱结果有可见提升
- 加成只在预期范围内生效
- 未触发奖励时，宝箱仍正常掉落

**Step 5: Commit**

```bash
git add src/modules/15-entities-base.js src/modules/13-game-loop.js src/modules/10-ui.js
git commit -m "feat: connect chest rewards to learning performance"
```

### Task 4：学习连胜与短时增益

**Files:**
- Modify: `src/modules/13-game-loop.js`
- Modify: `src/modules/10-ui.js`
- Modify: `src/modules/01-config.js`

**目标：**
让玩家持续保持“再答对一题就更强”的期待感。

**Step 1: 实现轻量学习连胜状态机**

定义清晰的连胜规则：
- 连续答对 3 次：短时掉落加成
- 连续答对 5 次：短时护盾或生命恢复
- 连续答对中断：降级而非直接全部清空

要求：
- 不做高复杂组合系统
- 不与已有战斗 buff 发生不可控叠加

**Step 2: 实现增益衰减与到期处理**

在主循环里统一处理：
- buff 剩余时间
- 到期提示
- HUD 状态清理

避免把计时逻辑散落到多个模块。

**Step 3: UI 反馈最小实现**

显示：
- 当前学习连胜层数
- 正在生效的学习奖励
- 即将达到的下一阈值

目标是“轻提示”，不是复杂面板。

**Step 4: 手工验收**

验证：
- 连胜推进逻辑稳定
- 到期后状态被正确清理
- 中断后不会产生负层数或脏状态

**Step 5: Commit**

- “答对啦！”

验证：

## 3. 风险与控制

### 3.1 风险：奖励过强，破坏平衡

**控制方式：**
- 先做短时奖励，不做永久强成长
- 宝箱只做加权，不做保底直升传奇
- 所有倍率放在配置里，方便热修或回调

### 3.2 风险：状态过多，代码变脏

**控制方式：**
- 所有奖励状态集中在一处维护
- 倒计时和 buff 统一在主循环处理
- 不在多个模块里复制计时逻辑

### 3.3 风险：低年级调优后太简单

**控制方式：**
- 只调前期体验，不全面移除挑战
- 低年级参数独立配置，不污染其他档位
- 把“更容易获得正反馈”与“完全无难度”区分开

## 4. 完成标准

满足以下条件，可视为第一阶段奖励闭环基本完成：

1. 学习行为能稳定触发即时奖励。
2. 宝箱或掉落能体现学习表现影响。
3. 学习连胜有轻量可见的收益。
4. 小学低年级前期挫败明显下降。
5. 原有主流程没有被破坏。
6. 文档与验收清单已补齐。

---

## 5. 建议执行策略

推荐按以下顺序推进：

1. Task 1：先搭配置与状态骨架
2. Task 2：先做答对后的即时奖励
3. Task 3：再接宝箱奖励绑定
4. Task 4：补学习连胜
5. Task 5：最后做低年级减压调优
6. Task 6：补文档、验证与回归记录

这样做的原因是：
- 先把基础结构搭好，后续不容易返工
- 先做最容易被玩家感知的即时奖励
- 再逐步把奖励扩到箱子、掉落和节奏层面
