# 单词库和数据库管理优化需求

> 创建时间：2026-03-04
> 状态：需求讨论中

## 需求理解

用户反馈当前项目的单词库管理和数据库管理存在以下问题：

### 核心痛点
1. **单词库管理混乱**：当前被描述为"灾难"
2. **数据库未充分利用**：有数据库但没有发挥应有作用
3. **缺少导入/导出功能**：用户无法方便地管理词库
4. **缺少数据迁移能力**：无法在设备间迁移学习数据
5. **学习进度统计不足**：无法方便地查看学习进度
6. **词库类型单一**：仅支持英语词汇，需要扩展到汉字、中文词汇、拼音等
7. **接口需要优化**：具体接口问题待明确

## 现状分析结果

### 1. 当前系统问题具体表现

根据深度分析（详见 [2026-03-05-vocabulary-system-analysis.md](2026-03-05-vocabulary-system-analysis.md)），已识别的核心问题：

**✅ "单词库管理是灾难"的具体表现**：
- ✅ 词库文件组织混乱：20+ 个词汇包，文件命名不统一，分卷逻辑复杂
- ✅ 词库加载逻辑脆弱：依赖全局变量 + 动态脚本加载，顺序敏感
- ✅ 词库更新困难：需要手动编辑 manifest.js，容易出错
- ✅ 词库格式不统一：部分使用分卷，部分使用完整包，重复定义

**✅ "数据库管理没利用起来"的具体表现**：
- ✅ 双重存储系统：LocalStorage（游戏使用） + SQLite（7.8MB，6544条词汇，未被使用）
- ✅ LocalStorage 容量瓶颈：5-10MB 限制，多账户场景容易溢出
- ✅ SQLite 数据库闲置：包含完整词汇数据但游戏代码未使用
- ✅ 缺少复杂查询能力：无法按难度/类别/时间段统计

#### 5.5 词库结构设计
**关键问题**：词库结构是否需要统一？

**方案 A：统一结构**（推荐）
- 优点：代码复用、易于维护、统一的学习体验
- 缺点：需要设计通用的数据结构，可能不够灵活
- 建议结构：
  ```javascript
  {
    id: "unique_id",
    type: "english|chinese_char|chinese_word|pinyin",
    content: "主要内容",
    translation: "翻译/释义",
    phonetic: "音标/拼音",
    metadata: {
      // 类型特定的元数据
      // 英语：phrase, imageURLs
      // 汉字：strokes, radical, components
      // 拼音：initial, final, tone
    }
  }
  ```

**方案 B：独立设计**
- 优点：灵活，每种类型可以有最优化的结构
- 缺点：代码重复、维护成本高
- 适用场景：不同类型差异很大时

**请选择**：[ ] 方案 A（统一） / [ ] 方案 B（独立） / [ ] 混合（基础统一，特殊字段独立）

- 当前问题：缺少复杂查询能力
- 优化方向：
