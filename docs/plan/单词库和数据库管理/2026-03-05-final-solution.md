# 词汇系统优化 - 最终落地方案（兼容现有架构版）

> 生成时间：2026-03-05  
> 文档状态：可执行（与当前项目代码对齐）  
> 目标定位：保持“老师可维护 CSV”的同时，不破坏现有运行时词库引擎

---

## 执行摘要

本方案保留你提出的核心方向：

1. 使用 `words/db/csv/entries.csv` 作为可编辑数据入口  
2. 提供教师可用的校验能力  
3. 为汉字/拼音扩展预留字段  
4. 优先解决稳定性和可维护性

但落地方式调整为：

- 不直接替换现有 `manifest + pack` 运行时模型
- 不在 `ensureVocabEngine()` 引入大范围异步改造
- 优先复用现有 `tools/vocab-db` 工具链（import/export/validate/audit/admin）
- 先做“兼容适配层”，再考虑引擎升级

---

## 为什么需要修订

原始方案中有几项与项目现状冲突：

1. 运行时目前依赖 `words/vocabs/manifest.js` + pack 文件加载  
2. 词库引擎 API 链路是同步风格，直接改成 CSV 异步加载会扩散改动  
3. 项目里已存在完整 `vocab-db` 工具链，重复建设会增加维护成本  
4. 使用 CDN 加载 PapaParse 不适合离线/校园网/APK 场景  
5. 当前真实数据中 `difficulty` 含 `external` 与空值，需按现状校验规则处理

---

## 最终架构（落地版）

### 数据与运行时分层

1. **编辑层（老师）**  
   - `words/db/csv/entries.csv`（Excel 可编辑）

2. **治理层（开发）**  
   - `tools/vocab-db/*`（import/export/validate/audit/publish）
   - `words/db/vocab.db`（治理数据库）

3. **运行时层（游戏）**  
   - `words/vocabs/manifest.js` + 各 pack JS（当前稳定路径）
   - `src/modules/09-vocab.js`（保留当前 pack 加载与轮换机制）

### 设计原则

1. 编辑源可以变，运行时接口尽量不变  
2. 先工具链治理，再运行时渐进迁移  
3. 所有新增能力优先“兼容模式”实现（可回滚）

---

## 模块划分（修订后）

### 模块 A：CSV 治理与发布（复用现有）

**范围**：`tools/vocab-db/*`、`words/db/*`  
**目标**：让 CSV 成为“可维护入口”，并稳定产出可运行数据

关键动作：

1. 统一使用 `npm run vocab:db:publish` 作为构建前门禁  
2. 校验规则按真实数据基线调整（非硬编码理想值）  
3. 产出审计报告供人工复核

---

### 模块 B：运行时兼容适配层（新增最小化）

**范围**：`src/modules/09-vocab.js`（小改）  
**目标**：不推翻现有引擎，逐步接入 DB 导出的数据

关键动作：

1. 保留 `ensureVocabEngine()` 现有模型  
2. 增加“可选数据源开关”（manifest 优先，DB 导出可灰度）  
3. 保证 pack 轮换、进度统计、自动词库逻辑不回归

---

### 模块 C：教师校验入口（工具复用，不重复造轮子）

**范围**：`tools/vocab-db/admin.html` + `validate.mjs`（增强）  
**目标**：提供老师可理解的校验结果与操作路径

关键动作：

1. 在现有 admin 页补“错误说明模板”和“修复建议”  
2. 输出可下载的错误清单（CSV/JSON）  
3. 若需纯静态页面，复用同一套规则，避免双规则分叉

---

### 模块 D：进度管理优化（在现有 storage 上增强）

**范围**：`src/storage.js`  
**目标**：解决容量与迁移，不引入平行进度系统

关键动作：

1. 完善 `saveJson/loadJson` 的压缩与回退策略  
2. 复用现有 `exportSaveCode/importSaveCode` 进行迁移  
3. 增加容量监控与提示（阈值告警）

---

## 分阶段实施计划（可执行）

## Phase 1（1-2 周，P0）：稳定与门禁

### 任务 1.1：数据门禁固化

1. 固化 `vocab:db:publish` 到发布流程  
2. 统一校验输出路径与报告命名  
3. 失败即阻断发布

### 任务 1.2：校验规则现实化

1. 将 `difficulty` 规则改为“白名单 + 可配置扩展”  
2. 对空字段分类处理（阻断项/警告项）  
3. 输出“老师可读”的错误文案

### 任务 1.3：storage 容量优化

1. 增强压缩路径（保持旧数据兼容）  
2. 增加容量占用检查  
3. 增加异常恢复日志

---

## Phase 2（1-2 周，P1）：运行时兼容接入

### 任务 2.1：可选数据源开关

1. 默认仍走 `manifest`  
2. 允许灰度加载 DB 导出数据（仅在开关开启）  
3. 出错自动回退 manifest

### 任务 2.2：回归测试

1. 词库选择 UI  
2. 自动轮换  
3. 进度累计与历史兼容  
4. 自定义词库导入链路

---

## Phase 3（2-4 周，P2）：多语言扩展准备

### 任务 3.1：数据模型扩展（非破坏）

1. 在 CSV 增加 `type`（english/hanzi/pinyin）  
2. 增加 `metadata`（JSON 字符串）  
3. 不影响英文既有字段读取

### 任务 3.2：渲染与练习模式试点

1. 先在单个 pack 试点汉字/拼音  
2. 通过 feature flag 控制  
3. 验证后再扩大范围

## 验收标准（修订后）

## 功能验收

1. 词库发布流程可一键执行并产出报告  
2. 游戏默认路径不变，功能无回归  
3. 数据源开关开启时可回退，且不崩溃  
4. 进度导入导出兼容旧存档

## 质量验收

1. `vocab:db:validate` 与 `audit` 可重复执行  
2. 错误报告可定位到具体行/字段  
3. 构建流程在校验失败时阻断

## 性能验收

1. 词库加载与当前版本同级或更优  
2. 不引入明显首屏阻塞  
3. 本地存储增长可监控、可告警

---

## 风险与缓解

### 风险 1：双路径维护复杂化

缓解：

1. 明确“manifest 为主，DB 导出为灰度”  
2. 单一校验规则源，避免校验分叉  
3. 每次发布保留回滚入口

### 风险 2：数据质量规则过严影响编辑效率

缓解：

1. 区分阻断错误与可接受警告  
2. 增加错误修复建议模板  
3. 保留批量修复脚本能力

### 风险 3：进度数据兼容问题

缓解：

1. 升级前自动备份  
2. 读路径兼容旧格式  
3. 导入失败给出可恢复提示

---

1. 先完成 Phase 1（不动运行时主链）  
2. 再做 Phase 2 灰度开关  
3. 最后做 Phase 3 多语言扩展
