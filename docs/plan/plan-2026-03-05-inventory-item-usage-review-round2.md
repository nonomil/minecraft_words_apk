## ⚠️ 新发现的问题

### 高优先级

#### 1. EmberParticle 类已存在但未说明复用
- **位置**：Plan 第 494 行
- **问题**：代码使用 `new EmberParticle()`，经验证该类已在 `15-entities-particles.js` 第 78-90 行定义
- **影响**：Plan 第 187 行说"新增火焰/传送粒子类"，但 EmberParticle 已存在，可能导致重复定义
- **建议**：修改 Plan 说明为"复用现有 EmberParticle 类，新增传送粒子类（SparkleParticle）"

#### 2. 粒子池机制未实现
- **位置**：Plan 第 187 行、第 234 行、第 265 行
- **问题**：多处提到"粒子池复用"，但第 494 行代码直接 `particles.push(ember)`，这不是池化机制
- **影响**：性能优化承诺无法兑现，可能导致大量粒子时性能下降
- **建议**：
  - 方案 A：删除"粒子池"相关描述，改为"限制粒子生成频率"（已在代码中实现：每 10 帧生成一次）
  - 方案 B：在 `15-entities-particles.js` 中实现简单的对象池（增加 ~30 行代码）

#### 3. Debuff 叠加未处理
- **位置**：Plan 第 468-481 行
- **问题**：`addDebuff` 方法直接 push 到数组，如果多个炸药同时爆炸，会添加多个 burn debuff，导致伤害倍增
- **影响**：游戏平衡性问题，玩家可以通过快速使用多个炸药秒杀敌人
- **建议**：在 `addDebuff` 中检查是否已存在同类型 debuff，如果存在则刷新 duration 而非叠加
```javascript
addDebuff(type, duration, params = {}) {
  // 检查是否已存在同类型 debuff
  const existing = this.debuffs.find(d => d.type === type);
  if (existing) {
    existing.duration = Math.max(existing.duration, duration); // 刷新持续时间
    return;
  }
  // ... 原有逻辑
}
```

#### 4. equippedConsumable 作用域问题
- **位置**：Plan 第 382-387 行（13-game-loop.js）、第 582-601 行（10-ui.js）
- **问题**：`equippedConsumable` 在 `13-game-loop.js` 中定义为全局变量，但 `10-ui.js` 在第 537 行加载（早于 13-game-loop.js 的第 560 行）
- **影响**：`updateConsumableUI()` 函数执行时 `equippedConsumable` 可能未定义，导致运行时错误
- **建议**：将 `equippedConsumable` 定义移到 `01-config.js` 中，确保在所有模块加载前初始化

### 中优先级

#### 5. 长按进度条动画重置不完整
- **位置**：Plan 第 347 行
- **问题**：`progress.style.animation = ''` 可能不足以重置 CSS 动画，在快速点击时可能导致动画卡住
- **影响**：用户体验问题，进度条可能显示不正确
- **建议**：添加强制重排
```javascript
progress.style.animation = '';
void progress.offsetWidth; // 强制重排
progress.style.animation = `hold-fill ${holdThreshold}ms linear`;
```

#### 6. 配置加载时机不明确
- **位置**：Plan 第 655 行
- **问题**：只说"在 bootstrap 初始化流程中调用"，但未说明是否需要 await，是否阻塞游戏启动
- **影响**：如果不 await，可能导致游戏启动时配置未加载完成
- **建议**：明确说明需要在 `17-bootstrap.js` 的 async 初始化函数中 await 该调用

#### 7. 改动量预估不一致
- **位置**：第 26 行说"420 行"，第 188 行表格显示"~490 行"
- **问题**：数据不一致，可能导致任务拆分判断错误
- **影响**：如果实际改动量接近 490 行，可能需要调整 worktree 拆分策略
- **建议**：统一为 490 行，并在修订摘要中说明原因

#### 8. 长按检测未处理多点触控
- **位置**：Plan 第 361-369 行
- **问题**：`onMove` 只检查 `e.touches[0]`，如果用户同时按下多个按钮（多点触控），可能导致误触发
- **影响**：移动端用户体验问题
- **建议**：添加 touch identifier 检查，确保只跟踪初始触摸点

### 低优先级

#### 9. 传送粒子类未定义
- **位置**：Plan 第 187 行说"新增火焰/传送粒子类"，但代码中只使用了 EmberParticle
- **问题**：末影珍珠使用 `end_particle` 类型（第 1057 行现有代码），但 Plan 未说明是否需要新增 SparkleParticle 类
- **影响**：如果配置中定义了 sparkle 粒子但类不存在，会导致运行时错误
- **建议**：明确说明复用现有 EndParticle 类，或在 `15-entities-particles.js` 中新增 SparkleParticle 类

#### 10. 雪球和岩浆桶未实现
- **位置**：验收标准第 283-284 行
- **问题**：配置示例（第 122-158 行）只包含炸药和末影珍珠，但验收标准要求雪球和岩浆桶
- **影响**：验收标准无法通过
- **建议**：在配置示例中补充雪球和岩浆桶的配置，或从验收标准中删除
