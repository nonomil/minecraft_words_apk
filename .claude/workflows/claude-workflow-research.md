# claude-workflow-research｜研究调研流程

> 本文档中的所有规则遵循 `claude-workflow-constants.md` 中的全局约束
> 触发条件：用户说"调研/对比/选型/搜索/研究/怎么实现/有哪些方案"
> 入口：从 `claude.md` 场景路由跳转至此
> 详细参考：`AI开发-PLan-Program-Debug-Claude和Codex协作/03-AI资料调研搜索阶段模板.md`

---

## 核心原则

- **先搜索，后结论**：不得在搜索完成前给出技术选型建议
- **多源交叉验证**：单一来源的结论不可信，至少 2 个独立来源
- **Codex 读文档**：利用 Codex 长上下文读取官方文档、源码、大型文件
- **CC 整合决策**：CC 负责对比分析和最终推荐，不做代码生成

---

## 调研深度分层（快速 / 标准 / 深度）

### 快速（默认）
- 触发：问题边界清晰、影响范围小、时间预算 < 1 小时
- 执行：本文件 Phase 0-3 即可

### 标准
- 触发：涉及技术选型，后续会进入复杂开发
- 执行：本文件完整 Phase 0-4，并产出 `docs/research/...` 报告

### 深度（强制切换）
- 触发任意一条：
  - 结论会影响架构主路径或长期维护成本
  - 方案存在明显分歧，快速/标准流程无法收敛
  - 需要多工具并行与 10 轮以上搜索迭代
- 执行入口：
  - `AI开发-PLan-Program-Debug-Claude和Codex协作/03-AI资料调研搜索阶段模板.md`
  - 按其 Phase 0-6 全流程执行，再回到本文件 Phase 3 进行结论汇总
  - 若外部模板不可达：执行本文件完整 Phase 0-4，并额外补充「风险 / 证据 / 回退方案」三段结论

## 研究流程（4阶段）

### Phase 0：问题定义（CC）

明确研究目标，不足则向用户提问：

```markdown
## 研究卡片

**研究问题：** [一句话描述要解决的问题]

**背景约束：**
- 技术栈：[语言/框架/版本]
- 不可改变的：[硬约束，如不能引入新依赖、必须兼容某版本]
- 性能/安全要求：[如有]

**期望输出：**
- [ ] 技术方案对比
- [ ] 推荐方案 + 理由
- [ ] 实现示例代码
- [ ] 风险点

**截止时间：** [如有]
```

---

### Phase 1：CC 快速搜索（WebSearch）

CC 先做快速搜索，建立基础认知：

```
WebSearch: "[技术关键词] best practices [当前年份]"
WebSearch: "[方案A] vs [方案B] comparison"
WebSearch: "[技术关键词] official documentation"
```

输出初步结论：
- 主流方案有哪几种
- 各方案的核心差异
- 需要深入研究的方向

---

### Phase 2：Codex 深度读取文档

**用 Codex 长上下文读取官方文档、GitHub README、大型配置文件。**

```
mcp__codex__codex({
  model: "gpt-5.4",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "
    ## Context
    - 研究问题：[Phase 0 的研究卡片]
    - 初步搜索结论：[Phase 1 的结论]

    ## Task
    深度研究以下内容，输出结构化摘要：

    1. 读取并分析：[文档URL / 文件路径 / 库名]
    2. 针对我们的约束条件（[约束列表]），评估每个方案的适用性
    3. 找出官方文档中关于 [具体问题] 的说明
    4. 列出每个方案的：
       - 核心优势
       - 已知限制
       - 适用场景
       - 不适用场景

    ## Acceptance
    - [ ] 每个方案有明确的适用/不适用判断
    - [ ] 结论基于官方文档，标注来源
    - [ ] 针对我们的约束条件给出明确建议
  "
})
// 保存 threadId
```

---

### Phase 3：CC 整合 + 交叉验证 + 输出结论

**Step 1：CC 整合 Phase 1 + Phase 2 结论**

对比矩阵：

```markdown
## 方案对比

| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
| 成熟度 | | | |
| 性能 | | | |
| 学习成本 | | | |
| 社区活跃度 | | | |
| 与现有栈兼容 | | | |
| 维护成本 | | | |

## 推荐方案
**推荐：** [方案X]

**理由：**
1. [理由1]
2. [理由2]

**排除其他方案的原因：**
- 方案A：[原因]
- 方案B：[原因]

**风险点：**
- [风险1] → 缓解方案：[方案]
```

**Step 2：Linus 三问验证推荐方案**
1. 这是真实问题还是想象的？→ 确认研究问题确实存在
2. 有没有更简单的方式？→ 是否有内置方案无需引入新依赖
3. 这会破坏什么？→ 引入推荐方案对现有代码的影响

**Step 3：向用户展示结论，等用户确认**

### Phase 4：输出调研报告（可选）

若用户需要留存，创建 `docs/research/YYYY-MM-DD-[topic].md`：

```markdown
# 调研报告：[topic]

**日期：** YYYY-MM-DD
**结论：** 推荐 [方案X]

## 背景
[研究问题和约束]

## 方案对比
[Phase 3 的对比矩阵]

## 推荐方案详述
[实现示例、关键配置、注意事项]

## 参考来源
- [来源1](URL)
- [来源2](URL)
```

```bash
git add docs/research/YYYY-MM-DD-[topic].md
git commit -m "docs: add research report for [topic]"
```

**必做：写入 `tasks/lessons.md`**
- 调研中发现的关键技术判断、常见误区、选型陷阱必须记录
- 格式参见 `claude-workflow-constants.md` 中的「Self-Improvement 全局规则」

## 研究完成后的衔接

- 结论明确 + 用户确认 → 根据任务规模选择后续流程：
  - 简单实现（≤3文件，≤200行）→ 回到 `claude.md` 简单模式
  - 复杂实现 → 以调研结论作为输入，进入 `claude-workflow-complex.md` Phase 1

---

## 止损规则

- 搜索 3 次仍找不到权威来源 → 向用户说明，请用户提供参考资料
- Codex 读取文档后结论仍模糊 → CC 缩小问题范围，重新调用 Codex
- 方案对比后无明显优劣 → 如实告知用户，列出权衡，由用户决策

### 1. 10 轮迭代搜索跟踪表

在 Claude 网页版（claude.ai）开启联网搜索，使用以下策略：
- 每轮搜索后审视当前结论的不足，判断是否正确，再发起下一轮
- 区分来源可信度：✅ 官方文档 / ⚠️ 社区经验 / ❌ 待确认
- 至少完成 10 轮后才输出最终结论

| 轮次 | 搜索关键词 | 新发现 | 修正了什么结论 | 仍不确定的问题 |
|------|-----------|--------|--------------|--------------|
| 1 | | | | |
| 2 | | | | |
| 3 | | | | |
| ... | | | | |
| 10 | | | | |

### 2. Google Deep Research 模式（生成搜索方向）

**工具：** Google 深度研究模式 / Gemini Deep Research

**用途：** 在 Phase 1 之后，生成系统性搜索方向作为多工具并行搜索的输入。

**Prompt 要点：**
```
主题：[RESEARCH_TOPIC]　目标：[RESEARCH_GOAL]
请输出：
1. 10-15 个核心搜索关键词（中英文各一套）
2. 5-8 个值得深入的子方向
3. 业内权威来源（官方文档、知名博客、学术论文）
4. 可能存在的争议点或常见误区
5. 推荐搜索顺序（从基础到深入）
```

将输出整理为"搜索方向清单"，分发给下一步的多工具并行搜索。

### 3. 多工具并行搜索

将 Google Deep Research 的搜索方向分发给多个工具，各自独立输出文档。

| 工具 | 适合场景 | 访问地址 | Prompt 要点 |
|------|----------|----------|------------|
| 秘塔 (Metaso) | 中文综合搜索 | metaso.cn | 粘贴搜索方向清单，要求引用官方来源、标注时间 |
| MiroMind | 深度研究报告 | dr.miromind.ai | 同上 |
| Perplexity | 英文技术文档 | perplexity.ai | 英文关键词搜索 |
| 天工 AI | 中文技术资料 | tiangong.cn | 中文关键词搜索 |
| Kimi | 长文档分析 | kimi.ai | 上传大型文档进行分析 |

各工具输出命名：`[OUTPUT_DIR]YYYY-MM-DD-[TOPIC]-[工具名].md`

### 4. Opus 多源文档整合（结构化报告）

**工具：** Claude Opus（Plan Mode）

将所有工具的输出文档交给 Opus 整合，要求：
- **结论一致性分析**：多源一致（高可信）/ 单一来源（需标注）/ 矛盾（需辩证）
- **落地性评估**：工程可行性（1-5分）、实施复杂度、维护成本、适用场景

**整合报告必须包含以下结构：**

| 章节 | 内容 |
|------|------|
| 执行摘要 | 3-5 句话核心结论，可直接用于决策 |
| 推荐方案 | 适用场景、核心优势、主要限制、落地步骤、可信度标记 |
| 备选方案 | 同上结构 |
| 矛盾点辩证分析表 | 争议点 / 来源A观点 / 来源B观点 / 最终判断 / 理由 |
| 仍不确定的问题 | 问题 + 建议验证方式 |
| 参考资料 | 来源 / URL / 可信度（官方/社区/个人）/ 时效性 |

保存到：`[OUTPUT_DIR]YYYY-MM-DD-[TOPIC]-final.md`

### 5. 结论验证实验

对调研报告中的关键结论，用 git worktree 做小规模实验验证：

```bash
# 根据 BRANCH_MODE 选择路径
git worktree add [WORKTREE_BASE]/research-verify -b research/verify-[TOPIC]   # temporary 模式
# 或
git worktree add research-verify -b research/verify-[TOPIC]                   # bare 模式
```

每个验证实验记录：
- **待验证结论** + 来源
- **最小可验证代码** + 预期结果 + 验证命令
- **实际结果**：✅ 成立 / ❌ 不成立 / ⚠️ 部分成立
- **修正**：若不成立，回溯修正调研报告

验证完成后清理 worktree。

### 7. 注意事项

1. **信息时效性**：AI 搜索结果可能包含过时信息，优先看官方文档日期
2. **来源标注**：每个结论必须标注来源，"AI 说的"不是来源
3. **矛盾是正常的**：不同工具给出矛盾结论时，不要取平均，要辩证分析
4. **实验验证优先**：调研结论再好，关键决策前必须小规模实验验证
5. **调研不是目的**：调研是为了做决策，超过 1 天没有结论就停止，用现有信息决策
6. **避免分析瘫痪**：信息足够做决策时就停止调研，完美是好的敌人
