# claude-workflow-debug｜Debug 流程

> 本文档中的所有规则遵循 `claude-workflow-constants.md` 中的全局约束
> 触发条件：用户描述 bug / 错误信息 / 测试失败 / 意外行为
> 入口：从 `claude.md` 场景路由跳转至此
> 详细参考：`AI开发-PLan-Program-Debug-Claude和Codex协作/06-AI调试阶段-复杂问题分析模板.md`

---

## 核心原则

- **先定位，后修复**：不得在复现和定位完成前提出修复方案
- **快速排查优先**：5 分钟内能定位则直接修复，不要用大炮打蚊子
- **Codex 读代码库**：利用 Codex 长上下文扫描大范围代码，CC 只看摘要
- **修复范围最小化**：只改导致 bug 的代码，不顺手重构周边

---

## 流程分层（Lite / Full）

本文件默认是 **Lite 版（5阶段）**，用于常规单点 Bug。

命中任意条件，必须切到 Full 版：
- 严重程度为 `P0/P1`
- 涉及 3 个及以上模块或明显跨模块状态同步（判定标准：修改文件来自 2 个以上的不同目录即视为跨模块）
- 需要数据迁移、接口兼容策略或大范围回归
- Lite 流程执行后仍无法稳定复现或无法收敛根因

Full 版入口：
- `AI开发-PLan-Program-Debug-Claude和Codex协作/06-AI调试阶段-复杂问题分析模板.md`
- 按其 `Phase 0-7` 完整执行（尤其是 Phase 3 架构分析、Phase 6 回归验证、Phase 7 根因文档）
- 若外部模板不可达：执行本文件 Lite 流程，并在 Phase 5 补齐根因文档与回归验证记录

## Debug 流程（Lite 5阶段）

### Phase 0：症状收集 + Bug 卡片（CC）

填写 Bug 卡片，信息不足则向用户提问：

```markdown
## Bug 卡片

**现象：** [用户/测试看到了什么]

**复现步骤：**
1. [步骤1]
2. [步骤2]

**预期行为：** [应该发生什么]
**实际行为：** [实际发生了什么]

**错误信息：**
[粘贴完整错误堆栈 / 控制台输出]

**环境：** 平台/版本/复现频率（必现 / 偶现）

**已尝试的方案：**
- [方案1] → 结果：

**影响范围：** 单用户 / 所有用户 / 特定条件
**严重程度：** P0 崩溃 / P1 功能不可用 / P2 体验问题
```

---

### Phase 1：快速排查（≤5 分钟）

先做最简单的检查，5 分钟内定位则直接跳到 Phase 4：

```bash
git log --oneline -10                        # 是否最近提交引入？
git log --oneline -5 -- [相关文件路径]       # 相关文件最近改动
git diff HEAD~3 -- [相关文件路径]            # 对比改动内容
```

快速排查清单：
- [ ] 是否是最近某次提交引入的？（可用 `git bisect` 定位）
- [ ] 是否有相关测试失败？
- [ ] 错误信息是否直接指向了问题代码？
- [ ] 是否是环境/配置问题（而非代码问题）？

**决策：**
- 5 分钟内定位 → 直接跳到 Phase 4 修复
- 未定位 → 进入 Phase 2

---

### Phase 2：Codex 扫描代码库定位根因

**用 Codex 长上下文读取相关代码范围，输出摘要给 CC。**

```
mcp__codex__codex({
  model: "gpt-5.3-codex",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "
    ## Context
    - 技术栈：[语言/框架/版本]
    - 错误信息：[完整错误/堆栈]
    - 相关文件范围：[用户指定 或 CC 初步判断的范围]

    ## Task
    扫描以上文件范围，定位此 bug 的根因。

    步骤：
    1. 读取所有相关文件
    2. 追踪错误堆栈中的调用链
    3. 列出所有可能导致此问题的原因（不要过早排除）
    4. 按可能性从高到低排序
    5. 找出导致错误的具体代码位置（文件名 + 行号）
    6. 分析根因：是逻辑错误、边界条件、数据问题还是依赖问题？

    ## Acceptance
    - [ ] 明确指出根因文件和行号
    - [ ] 说明触发条件（什么输入/状态会触发）
    - [ ] 说明影响范围（只影响此处还是有其他调用方）
    - [ ] 给出置信度（高/中/低）
  "
})
// 保存 threadId
```

记录 Codex 结论：
```
根因：[文件:行号] [一句话描述]
触发条件：[描述]
影响范围：[描述]
置信度：高 / 中 / 低
建议修复：[方案]
```

---

### Phase 3：CC 制定修复方案 + 交叉验证 + 用户确认

**Step 1：CC 基于 Codex 摘要制定修复方案**

```
根因：[文件:行号] [一句话描述]
触发条件：[描述]
影响范围：[描述]

修复方案：
- 修改文件：[路径]
- 改动描述：[具体改什么]
- 预估 diff：[行数]
- 是否影响现有 API：[是/否，如是说明]

验证方式：
- [ ] 复现步骤执行后不再报错
- [ ] 相关测试通过：[测试命令]
- [ ] 无回归：[全量测试命令]
```

**Step 2：交叉验证（Codex 置信度低 或 涉及多模块时执行）**

若 Codex 置信度为"低"，或问题涉及多模块交互，CC 用 Plan Mode 做架构层分析：

```
[CC Plan Mode]
请从架构角度分析此 bug：
- 问题发生在哪个模块边界？
- 是否是状态同步 / 时序 / 接口约定问题？
- Codex 认为根因是 [X]，你是否同意？
- 最小修复 vs 根本修复的权衡？
```

结论对比：
- 两者一致 → 直接进入 Phase 4
- 有分歧 → 加临时日志验证，用数据决定；仍不确定 → 采用最小修复先止血

**Step 3：向用户展示方案，等用户确认后继续**

---

### Phase 4：Codex 执行修复 + Review

**Step 1：先写复现测试（确认失败）**

```bash
# 写能复现 bug 的测试，运行确认失败
npm test -- --grep "[Bug 复现测试名]"
# 预期：FAIL
```

**Step 2：Codex 执行修复**

```
mcp__codex__codex_reply({
  threadId: "<Phase 2 的 threadId>",
  prompt: "
    ## Task
    按以下修复方案修改代码：
    [CC 制定的方案]

    ## Constraints
    - 范围：仅修改 [指定文件]，不得改动其他文件
    - 最小修改：只改导致 bug 的代码，不重构周边
    - API：不得修改现有函数签名
    - 清理：移除所有临时调试日志

    ## Acceptance
    - [ ] 复现测试通过
    - [ ] 全量测试无回归
  "
})
```

**Step 3：两轮 Review**
- 轮次1（CC）：修复是否正确？是否引入新问题？是否最小修改？
- 轮次2（Codex）：边界条件？其他调用方是否受影响？
- 两轮一致无新问题 → 向用户汇报，等用户确认

**Step 4：清理调试日志 + 提交**

```bash
grep -r "\[DEBUG\]" src/   # 确认无遗留调试日志

git diff --stat            # 确认 diff ≤ 200 行
git add [修改的文件]
git commit -m "fix: [一句话描述根因和修复]"
```

### Phase 5：根因文档记录与复盘

**5.1 根因文档记录（可选但推荐）**

创建 `docs/debug/[bug-id].md`：

```markdown
# Bug 记录：[bug-id]

**日期：** YYYY-MM-DD  **状态：** ✅ 已修复

## 现象
[SYMPTOM]

## 根本原因
[最终确认的根因，一句话]

## 分析过程
- Codex 结论：[结论] / 置信度：[高/中/低]
- CC 交叉验证：[结论]
- 最终判断：[结论]

## 修复方案
- 文件：`[修改的文件]`
- 改动：[描述]
- Commit：`[commit hash]`

## 预防措施
- 新增测试：`[测试文件路径]`
- 后续注意：[需要注意的点]
```

```bash
git add docs/debug/[bug-id].md
git commit -m "docs: add debug record for [bug-id]"
```

**5.2 同步更新 `tasks/lessons.md`（强制必做）**：
```
- [YYYY-MM-DD] [bug-id/场景] → [根因一句话] → [下次防范方式]
```

> 此步骤与 `CLAUDE.md` 的自我改进循环联动，确保同类 Bug 不再复现。

## 止损规则

- Codex 扫描后仍无法定位根因 → CC 缩小范围，重新调用 Codex
- 修复后引入新 bug → 回到 Phase 2 重新定位
- 修复涉及文件 > 3 或 diff > 200 行 → 升级为复杂模式（跳转 `claude-workflow-complex.md` Phase 1，将已定位的根因作为已知背景）
- Codex 连续失败 3 次 → CC 接管，手动分析后重新提供更精确的 Prompt

## 补充：完整调试扩展（参考 reference/07）

> 以下内容从 `reference/07-复杂问题调试模板.md` 提取，补充 Lite 版缺失的关键环节。
> 切到 Full 版时，这些内容为必选项。

### 1. Phase 3 扩展：CC Plan Mode 架构分析（完整版）

Lite 版 Phase 3 仅做简要交叉验证，Full 版需覆盖以下五个维度：

**系统边界分析：**
- 数据流：数据如何在模块间流动？谁生产、谁消费、谁变换？
- 状态管理：哪些模块持有相关状态？状态是否有多份拷贝？

**根因假设（四类排查）：**

| 假设类型 | 排查要点 |
|----------|----------|
| 接口设计问题 | 模块间接口契约是否一致？参数/返回值类型是否匹配？ |
| 状态同步问题 | 多处持有同一状态时，更新是否同步？是否存在脏读？ |
| 时序/异步问题 | 异步操作的执行顺序是否有保证？竞态条件？ |
| 数据格式不一致 | 上下游对数据结构的约定是否一致？序列化/反序列化是否对称？ |

**Codex 结论验证：**
- Codex 认为根因是 [X]，CC 是否同意？
- 不同意则给出 CC 自己的假设及依据

**修复策略权衡：**

| 策略 | 适用场景 | 风险 |
|------|----------|------|
| 最小修复（治标） | 线上紧急、需快速止血 | 根因未消除，可能复发 |
| 根因修复（治本） | 有充足时间、问题反复出现 | 改动范围大，回归风险高 |

**影响评估：**
- 修复后需回归测试哪些功能模块？
- 是否涉及数据迁移需求？

### 2. Phase 4 扩展：交叉验证对比表 + 决策规则

**结论对比表（必填）：**

| 维度 | Codex 结论 | CC/Opus 结论 | 一致？ |
|------|-----------|-------------|--------|
| 根本原因 | | | |
| 修复方案 | | | |
| 影响范围 | | | |
| 修复优先级 | | | |

**决策规则：**
- 两者一致 → 直接进入修复阶段
- 两者不一致 → 执行验证实验（加临时日志 / 写针对性单测），用数据决定
- 验证实验后仍不确定 → 采用"最小修复"策略，先止血再根治

**验证实验模板：**
```javascript
// 在关键路径加临时日志
console.log('[DEBUG][BUG_ID] 关键变量:', {
  变量名: 变量值,
  时间戳: Date.now()
});
```

### 3. Phase 6 补充：回归验证（Lite 版缺失）

Lite 版无独立回归验证阶段，Full 版要求在修复提交后执行：

```bash
# 1. 运行受影响模块的测试
npm test -- --grep "[受影响模块]"

# 2. 手动验证关键路径
# - [手动验证步骤1]
# - [手动验证步骤2]
```

**检查清单：**
- [ ] 受影响模块测试全部通过
- [ ] 手动验证关键用户路径通过
- [ ] 无新增 warning 或 deprecation

### 4. Phase 7 扩展：根因文档"经验教训"部分

Lite 版 Phase 5 的根因文档缺少"经验教训"，Full 版要求追加：

```markdown
## 经验教训
[这个 Bug 教会了我们什么？如何避免类似问题？]

示例：
- 跨模块传递数据时，必须在边界处做 schema 校验
- 异步操作链中，每一步都应有超时和错误兜底
- 序列化/反序列化必须有对称的单元测试覆盖
```
