# Git 版本管理

根据参数执行不同操作。无参数时根据对话上下文智能判断。

## 子命令

### /git save - 提交完成的修改

**先判断是否应该提交**：

| 应该提交 | 不应该提交 |
|---------|-----------|
| 完成一个完整功能 | 只是尝试性修改，还没确定方案 |
| 修复了 bug 并验证通过 | 修改后问题仍未解决 |
| 用户明确说"完成了"、"可以了" | 中间状态，还需继续修改 |
| 重构完成，功能正常 | 调试代码、临时修改 |

**如果判断不应该提交**：不执行，继续当前任务。

**如果判断应该提交**，执行以下步骤：
1. `git status --porcelain` 检查变更
2. 分析修改内容，生成提交信息（Conventional Commits 格式）
3. 只添加代码文件：`git add <代码文件>` （不添加 CLAUDE.md、配置文件、文档等）
4. `git commit -m "message" --no-verify`
5. 检查是否需要更新 CLAUDE.md（架构变更、新增模块等），单独更新但不提交
6. 告知用户：已提交 + 哈希值

### /git checkpoint - 保存临时状态
**触发条件**：开始复杂任务前、暂停工作、尝试风险操作前

步骤：
1. `git add -A && git commit -m "WIP: [当前任务描述]" --no-verify`
2. 告知用户：检查点已创建 + 哈希值

### /git history [文件/关键词] - 查看历史
**触发条件**：新对话需要了解之前改了什么、不理解某段代码来源

步骤：
1. `git log --oneline -20` 或 `git log --oneline -10 -- <file>`
2. 如需详情：`git show <hash>`
3. 总结相关修改历史

### /git restore <hash> [file] - 还原版本
**触发条件**：多次修改仍失败、用户要求回退、改坏了

步骤：
1. 确认还原目标（整个提交还是某个文件）
2. `git checkout <hash> -- <file>` 或 `git revert <hash>`
3. 告知用户：已还原

### /git status - 查看当前状态
步骤：
1. `git status`
2. `git diff --stat`（如有未提交修改）
3. 总结当前状态

## 智能判断（无参数时）

根据对话上下文判断应该执行哪个操作：

| 场景 | 判断依据 | 动作 |
|------|---------|------|
| 应该提交 | 用户说"完成了"、"可以了"、测试通过 | save |
| 应该查看历史 | 用户问"之前改了什么"、新对话、不理解代码 | history |
| 应该还原 | 用户说"回到之前"、"还原"、多次失败 | restore |
| 应该创建检查点 | 即将开始复杂修改、用户说"先保存一下" | checkpoint |

## 注意事项
- 只提交代码文件，不提交其他文件（如 CLAUDE.md、配置文件、文档等）
- 不要提交 .env、密钥等敏感文件
- checkpoint 用于临时保存，save 用于正式提交
- 更新 CLAUDE.md 时只更新相关部分，不要重写整个文件
