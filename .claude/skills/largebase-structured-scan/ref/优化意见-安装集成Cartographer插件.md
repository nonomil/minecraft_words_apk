## 业界实际在怎么做

研究表明，代码是逻辑结构化的，不同于非结构化文本。批评 RAG 的人认为，高级工程师熟悉新项目的方式是探索文件夹结构、追踪 import 语句、通读整个文件来建立架构心智模型——而不是靠向量相似度检索破碎的代码块。

这个批评直接命中你的场景。主流方案目前分三个流派：

**流派1：纯 Markdown 分层索引（最适合你的场景）**

一个被验证有效的方法是：主 agent 只读 50 行的 registry 文件，而不是 5000 行的完整报告。核心结构是 `.claude/reports/_registry.md` 作为索引入口，各子目录存放分类报告（arch/、handoff/、impl/ 等）。这个方案把"扫描一次、多次复用"做到了极致。

**流派2：子 agent + 持久 memory**

Claude Code 的 subagent memory 字段给子 agent 提供跨会话的持久目录，agent 在这里积累代码库模式、调试洞察和架构决策。系统自动在 MEMORY.md 前 200 行注入持久记忆，超过时 agent 自行整理压缩。

**流派3：PROJECT_INDEX.json + 并行子 agent 扫描**

Claude Code Project Index 工具自动生成 `PROJECT_INDEX.json`，包含项目结构、函数、类和调用关系，支持多种编程语言。另有 `bootstrap-repo` 命令用 10 个并行子 agent 搜索代码库生成综合文档。还有 `tldr-code` 提供 5 层代码分析，支持树形结构、代码结构、控制流图、数据流图、影响分析，专为给 LLM 提供 token 高效的代码摘要而设计。

## 最值得参考的是：**Cartographer**

Cartographer 的工作流是：先用 `scan-codebase.py` 递归扫描文件树并统计每个文件的 token 数，然后按模块分组、平衡 token 预算（目标每个 agent ~500k tokens），接着并行派发多个 Sonnet subagent 各自读取分配到的文件、分析用途和依赖，最后合并所有报告生成架构图和导航指南，输出 `docs/CODEBASE_MAP.md` 并自动更新 CLAUDE.md。

这个设计和你想要的几乎完全一致，而且它解决了你当前方案最大的问题——**产物直接写入 CLAUDE.md，每次会话零成本获取上下文**。

更关键的是它的配套工具 codex-orchestrator 说明了这个设计的真实价值：给每个 Codex agent 的 prompt 注入 `docs/CODEBASE_MAP.md`，就能让 agent 立刻理解整个代码库的文件用途、模块边界、数据流和依赖关系，而不是浪费时间盲目探索结构。
