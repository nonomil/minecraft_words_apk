# 工作流文档体系 全量审查报告

> 审查范围：CLAUDE.md + 7个 workflow 子文档
> 审查视角：跨文件一致性 + 结构性问题 + 可执行性

---

## 一、跨文件一致性问题（最高优先级）

### 1. `reasoning` 参数：constants 有规则，调用处没落地

**现状**：
- `constants.md`："Codex 推理强度规则"已定义 high/medium 两档，Prompt 约束语句也写了
- `CLAUDE.md` 必填参数块：**没有 `reasoning` 字段**
- `complex.md` / `debug.md` / `research.md` 的 Codex 调用示例：**全部缺失 reasoning 参数**

**影响**：规则定义在 constants，执行在各 workflow，两者脱节，每次调用都会遗漏。

**修复**：在 constants 的必填参数模板中加入 `reasoning`，各 workflow 的调用示例同步更新。

---

### 2. `lessons.md` 写入时机：constants 定义 3 条，各文件实际执行不一致

**现状**：
- `constants.md` Self-Improvement 区块写了 3 个触发事件
- `debug.md` Phase 5：✅ 有"同步更新 tasks/lessons.md（必做）"
- `complex.md` Phase 6 末尾：✅ 有写入提醒
- `research.md` Phase 4：❌ 写的是"**可选**写入 lessons.md"——与 constants 的"强制"矛盾
- `parallel.md` Phase 10.4：❌ 写的是"**推荐**"——同样与"强制"矛盾
- **CLAUDE.md 强制门禁**：lessons.md 读取放在括号里"若存在"——看起来可选

**修复**：
1. `research.md` Phase 4 和 `parallel.md` Phase 10.4 改为"必须"
2. CLAUDE.md 强制门禁中的 lessons.md 读取单独列为步骤，去掉括号

---

### 3. 路由优先级：constants 和 CLAUDE.md 有，README 也有，三处措辞略有出入

**现状**：优先级在三处都有定义（constants / CLAUDE.md / README），内容基本一致，但：
- CLAUDE.md 场景路由表的**行顺序**是"简单→复杂→Debug→并行→调研→大型"（正序）
- 优先级列表是"Debug→调研→大型→复杂→简单"（倒序）
- 两者顺序相反，CC 扫表时容易按表顺序判断

**修复**：CLAUDE.md 场景路由表按优先级从高到低重排，与优先级列表保持一致。

---

### 4. 模式升级触发条件：只在 CLAUDE.md 简单模式里，复杂模式里没有对应机制

**现状**：
- CLAUDE.md 简单模式有"⚠️ 模式升级触发条件"
- `complex.md` 没有对等的"任务拆分触发条件"——复杂模式 Review 超 5 轮才在补充里提到止损阈值，但没有统一的升级/降级机制

**修复**：在 constants 里统一定义"止损与升级阈值"，简单/复杂模式都引用同一份。

---

### 5. 验证完成门禁：constants 有一份，CLAUDE.md 简单模式也有一份，措辞略有差异

**现状**：
- `constants.md` Self-Improvement 末尾："验证完成门禁（全 workflow 通用）"3条
- `CLAUDE.md` 简单模式 Step 3 验证门禁：也是 3 条，但第 3 条多了"有空则重构"
- `complex.md` 补充第 5 条：再次列了同样的 3 条

**影响**：三处各说各的，CC 不知道哪个是"权威版"。

**修复**：constants 版作为权威，其余两处改为"参见 constants 中的验证完成门禁"。

## 二、结构性问题

### 6. CLAUDE.md 模式升级触发条件位置错误

**现状**：升级条件夹在简单模式 Step 3 和 Step 4 之间，CC 容易理解为"Step 3 完成后才检查一次"。

**修复**：提到简单模式流程开头，加注"全程持续检查，任意步骤触发立即处理"。

---

### 7. `research.md` 调研深度分层入口不清晰

**现状**：
- 文件顶部定义了"快速/标准/深度"三层
- 但在 Phase 0-4 流程里没有分叉点——CC 不知道何时进入深度路径
- 深度路径触发条件在开头写了，但流程里没有对应的"if 深度 → 跳转"标记

**修复**：在 Phase 1 结束时加一个显式决策节点：
```
→ 判断深度等级：
  快速 → 继续 Phase 2
  标准 → 继续 Phase 2，Phase 4 必须产出报告
  深度 → 进入深度调研扩展（补充第1-7节）
```

---

### 8. `parallel.md` 补充的发布流程（Phase 0-10）与主流程（Phase 1-3）编号冲突

**现状**：主流程是 Phase 1/2/3，补充的发布流程是 Phase 0-10，两套编号独立存在于同一文件，混淆严重。

**修复**：补充发布流程改为"附录：完整发布流程"，与主流程 Phase 分开命名。

---

### 9. `largebase.md` Step 0（Cartographer）和 Step 1（API 成本提示）顺序存在逻辑问题

**现状**：Step 0 是"可选增强"，Step 1 是"强制 API 成本提示"——但 Step 1 的触发需要先知道文件数（来自 Step 0 的扫描结果）。实际上 Step 1 必须在 Step 0 之后，但文档描述里 Step 1 的触发条件没说明数据来源。

**修复**：Step 1 说明"文件数来自 Step 0 的统计结果或 CC 预估"，明确数据依赖。

---

### 10. `init.md` 与 constants 的"项目配置"区块职责重叠但不互相引用

**现状**：
- `init.md` 定义了配置写入流程
- `constants.md` 有"项目配置"区块存储结果
- 两者存在但没有互相引用——CC 运行 init 后可能不知道要写到 constants，或者 constants 的配置区块不知道由 init 生成

**修复**：`init.md` Step 7 明确引用 constants 的"项目配置"区块路径；constants 的项目配置区块注明"由 init.md 流程写入"（已有一句，但 init.md 那侧缺少对应指针）。

## 三、可执行性问题

### 11. `complex.md` Phase 5（Opus 审查）跳过条件写法混乱

**现状**：
```
（满足任一条件可跳过：Plan 仅含 <3 个 worktree；Phase 3 连续2轮无新问题；用户明确说"不需要 Opus 审查"）
```
三个条件写在括号里，且第一个（<3 worktree）本身不足以说明 Opus 是否必要。

**修复**：改为独立小节，明确"Phase 5 默认执行，以下情况可跳过"。

---

### 12. `debug.md` Lite/Full 分层切换条件不够机械

**现状**：Full 版触发条件写了 4 条，但"涉及 3 个及以上模块"和"明显跨模块状态同步"语义重叠，CC 判断时可能主观。

**修复**：补充更具体的判断标准，例如"修改文件来自 2 个以上目录 = 跨模块"。

---

### 13. 各 workflow 的"止损规则"格式不统一

**现状**：
- `research.md`：3 条散文式
- `debug.md`：4 条散文式
- `complex.md`：止损阈值在补充第 4 条里，是表格式
- `parallel.md`：3 条散文式

**修复**：统一格式（推荐表格：触发条件 | 动作），放在每个文件末尾固定位置，便于 CC 快速查阅。

---

### 14. `CLAUDE.md` 中 Codex Prompt 模板的 `推理强度` 写法与 constants 定义不同

**现状**：
- CLAUDE.md Prompt 模板：写的是"审查/调试/规划：`需要深度推理，先分析再结论。`"
- constants 推理强度规则表：有明确的"high/medium"档位 + 对应 Prompt 语句
- 两者措辞不同，规则拆在两处

**修复**：CLAUDE.md 的 Prompt 模板中推理强度部分，直接引用 constants 的规则表，去掉重复定义。

### 17. 各文件顶部声明格式不统一

**现状**：所有 workflow 文件顶部都有引用 constants 的声明，但格式各不相同：

```

> 触发条件：[条件]
> 入口：从 claude.md 场景路由跳转至此

```

### 18. constants.md 必填参数模板仍缺 `reasoning` 字段（与 #1 相关但方向不同）

### 22. `init.md` Step 6 Bare Repo 初始化脚本含 `rm -rf` 禁止命令
