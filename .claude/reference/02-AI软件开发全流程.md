## 工具角色定义

| 工具 | 角色 | 擅长 | 何时用 |
|------|------|------|--------|
| Antigravity + Opus + Plan Mode | 需求分析师 + 架构师 | 深度需求澄清、架构决策、发现歧义 | **仅用于最初的 Plan 生成**，质量最高 |
| Claude Code for VS Code + Plan Mode | 开发中的小计划 | 单任务分析、文件搜索、改动预览 | 日常开发中每个任务前 |
| Codex (gpt-5-codex) 插件版 | 高级工程师 | step-by-step 计划、代码生成、长时间连续执行 | 代码生成、工程审查、MCP 自动调用 |
| Claude Sonnet (Kiro/Antigravity) | 快速编码 | 中等复杂度任务、快速迭代 | 并行开发分支二 |
| Trae + gpt-5-codex | 省 Token 编码 | 简单任务、节省成本 | 并行开发分支三（需频繁点继续） |

**为什么 Antigravity + Opus 的初始 Plan 质量最高：**
- Opus 有 1M token 上下文，长程连贯性最强，不会在复杂计划中丢失细节
- Plan Mode 强制只读探索，先理解再行动，防止过早写代码
- Opus 会主动发现需求歧义并要求澄清，而不是猜测填充
- SWE-bench 74.5%，真实代码库架构理解最深

## 日常开发环境设置

```
VSCode 集成插件  →  Claude Code for VS Code + Codex 插件  →  可视化/终端开发
VSCode 侧边栏    →  Claude Code and Codex Assist (agsoft.claude-history-viewer)
                 →  同时查看 Claude/Codex 历史对话、看 Diff、复制 Session ID 恢复会话
Cloud CLI（Claude Code UI）→ Claude Code/Cursor CLI/Codex 的桌面端与移动端界面
                            → 可视化对话/终端对话/多窗口并行
                            → https://github.com/siteboon/claudecodeui
                            → 可在 IDE/浏览器中打开多个同页面，并行查看多个对话进展
```

两者读写同一套 `~/.claude/projects/*.jsonl`，天然互通，无需额外配置。

**轻量协作（无 MCP）：**

```bash
# 场景A：速度优先 — Codex 先实现，Claude Code 审查
codex "实现 xxx 功能"
claude > 审查 src/xxx.ts，检查边界条件和安全隐患，有问题直接修

# 场景B：质量优先 — Claude Code 先规划，Codex 填充
claude > 设计 xxx 模块的架构，输出接口定义
codex "按照 src/xxx.ts 的接口定义，补全实现"

# Git 作为交接接口（推荐）
git commit -m "codex: initial implementation"
claude > git diff HEAD~1 看 Codex 的改动，审查并修复，完成后提交
```

## AI 代码质量现实与应对策略

> 基于实践数据（2026Q1）：AI 首轮代码质量约 60-70 分，每轮 Review 只能召回部分问题。

**核心原则：**
- 单次变更 diff ≤ 200 行，超过则拆分（大变更是 Review 9 小时的根因）
- 至少 3 轮 Review（CC 快速审查 + Codex 深度审查各自独立，第3轮 Dev Agent 验证），连续 2 轮无新问题才提交（例外：diff ≤ 50 行的微小改动可 2 轮，需明确标注）
- Review Agent（Codex）找候选问题，Dev Agent（Claude Code）结合需求 context 验证
- Context 压缩会导致质量下降，长任务定期刷新 Session

**Review 工具优先级（按效果排序）：**

| 工具 | Review 质量 | 速度 | Token 消耗 | 推荐用途 |
|------|------------|------|-----------|---------|
| Codex 超高推理 | ⭐⭐⭐⭐⭐ | 慢 | 高 | 深度审查，找长尾问题 |
| Claude Code | ⭐⭐⭐⭐ | 中 | 中 | 快速审查，结合需求验证 |
| Cursor Review Agent | ⭐⭐ | 快 | 低 | 不推荐，误报率高，价值有限 |

**Review 预算规划（按任务规模）：**

| 任务规模 | 预期 Review 轮次 | 预期额外时间 |
|---------|----------------|------------|
| diff ≤ 50 行（微小改动，需标注） | 2 轮（例外） | 短 |
| diff 50-200 行 | 3 轮 | 中 |
| diff > 200 行 | 拆分！不要 Review | — |
| 大重构（5k+ 行） | 不要一次做，必须拆分 | — |

**Context 刷新策略：**
- 单个 Codex Session 执行超过 3 个任务 → 开新 Session
- 发现明显低质量输出（逻辑混乱、忽略约束）→ 立即重启
- 长时间运行后（>2小时）→ 重启 Session，重新提供 context

## 自动化 vs 手动控制总览

```
调研阶段    手动（多工具并行搜索）→ 手动（Opus 整合判断）
计划阶段    手动（Opus 需求澄清）→ 手动（交叉 Review）→ 可自动（Codex 生成 steps）
开发阶段    手动（Plan Mode 确认）→ 自动（Codex MCP 执行）→ 手动（验收）
调试阶段    手动（问题分析）→ 手动（根因确认）→ 自动（Codex MCP 修复）→ 手动（验收）
```

| 阶段 | 步骤 | 控制方式 | 理由 |
|------|------|----------|------|
| 计划 | Opus 需求澄清 | 手动 | 歧义判断需人介入 |
| 计划 | Codex 工程审查 | 手动 | 需人决定采纳哪些意见 |
| 计划 | 交叉 Review | 手动 | 终止条件是主观判断 |
| 计划 | Codex 生成 step-by-step | **可自动** | 输入确定，格式固定 |
| 开发 | Plan Mode 分析 | 手动 | 需人确认改动范围 |
| 开发 | Codex MCP 代码生成 | **自动** | 已通过 MCP 集成 |
| 开发 | 测试/提交 | 手动触发 | 每次 Codex 输出需先经 Claude 验收，未验收禁止提交 |
| 调试 | 问题收集/快速排查 | 手动 | 需人描述现象 |
| 调试 | Codex 超高推理分析 | 手动触发 | 需人提供完整上下文 |
| 调试 | 根因确认 | 手动 | 最关键的判断节点 |
| 调试 | Codex MCP 修复实施 | **可自动** | 根因确定后格式化任务 |
| 调试 | 回归验证 | 手动 | 需人确认修复符合预期 |

## 阶段一：计划阶段（Plan Phase）

### Step 1：需求澄清（Claude Opus × Plan Mode）

打开 Antigravity，切换到 **Plan Mode + Opus 模型**，输入需求描述：

```
我需要实现 [功能描述]。
请帮我：
1. 澄清需求中的歧义点
2. 识别技术风险
3. 输出一份 Plan 方案文档，保存到 docs/plan/YYYY-MM-DD-[feature].md
```

Plan 文档结构：
```markdown
# [功能名] Plan 方案

## 需求理解
[Opus 对需求的理解，含边界条件]

## 技术方案
[架构选择、关键设计决策]

## 风险点
[潜在问题、依赖、不确定项]

## 待确认问题
[需要用户回答的问题]
```

完成后：将文档移动到当前项目的 `docs/plan/` 文件夹。

**任务解耦完成定义（必须同时满足）：**
- 每个任务有独立输入/输出与独立验收标准。  
- 单任务可以独立执行并独立验证，不依赖“隐式上下文”。  
- 任务依赖关系可画成无环有向图（DAG），无循环依赖。  
- 并行任务不同时修改同一关键文件；若无法避免，需提前标注冲突点与合并策略。

- [ ] Plan 文档已生成并移入 `docs/plan/`

---

### Step 2：Codex 审查 Plan（Codex × gpt-5-codex）

打开 Codex 插件版（gpt-5-codex），输入：

```
请审查这份 Plan 文档：[docs/plan/xxx.md]
从工程实现角度：
1. 方案是否可行？
2. 有哪些遗漏的技术细节？
3. 步骤顺序是否合理？
4. 有哪些需要修改的地方？
5. 任务是否已经充分解耦（边界清晰、依赖无环、可并行执行）？
```

根据 Codex 反馈修改 Plan 文档。

- [ ] Codex 审查完成，Plan 已修改

---

### Step 3：交叉 Review（Opus ↔ Codex，≥3 轮）

**轮次规则：**
- Opus 分析 → Codex 审查 → Opus 再分析 → 直到双方无疑问
- 至少 3 轮，连续 2 轮无新问题才收敛；超过 5 轮仍有重大分歧 → 停止，重新澄清需求

```
[Opus] 请再次分析 Plan 文档，Codex 提出了以下修改意见：[意见列表]
       你是否同意？有无补充？

[Codex] 请审查修改后的 Plan 文档，确认是否还有工程问题。
```

- [ ] 双方无疑问，Plan 文档已定稿

---

### Step 4：生成 Step-by-Step 开发计划（Codex）

Plan 定稿后，让 Codex 生成详细执行计划：

```
基于 Plan 文档 [docs/plan/xxx.md]，请生成 step-by-step 开发计划：
- 每个任务：写失败测试 → 运行确认失败 → 实现最小代码 → 运行确认通过 → 提交
- 每步包含：确切文件路径、代码片段、验证命令、预期输出
- 标注每个任务的依赖任务（若无则写 none）与并行标签（serial/parallel）
- 输出到：docs/development/YYYY-MM-DD-[feature]-steps.md
```

- [ ] Step-by-step 开发计划已生成

---

### Step 4.5（可选）：Opus 交叉 Review 开发计划

```
[Opus Plan Mode] 请审查这份开发计划：[docs/development/xxx-steps.md]
重点检查：
1. 任务粒度是否合理（每步 2-5 分钟）？
2. 测试设计是否覆盖边界条件？
3. 任务间依赖关系是否正确？
```

- [ ] 开发计划已通过 Opus Review（可选）

### Step 4.6（强制）：人工解耦复核门禁

进入并行开发前，人工复核一次“任务解耦”：
- [ ] 每个任务均有独立验收标准  
- [ ] 依赖图无环，顺序可执行  
- [ ] 并行任务文件改动边界清晰  
- [ ] 冲突预案明确（冲突负责人 + 合并策略）

未通过复核，不得进入 Step 5。

## 阶段二：开发阶段（Development Phase）

### Step 5：并行 Worktree + Codex 开发

**完整并行流程：**

```
Opus 写好 TDD 要求 + 解耦任务（Phase 4.5 已完成）
  ↓
按解耦后的并行批次，开多个 worktree
  ↓
每个 worktree 分配一个 Codex 实例，各自独立执行
  ↓
每个 worktree：Codex 执行 → CC 验收（步骤 3.4.5）→ 提交
  ↓
批次完成后，Opus 合并所有分支，解决冲突
  ↓
e2e 测试验证整体功能
```

**创建 worktree：**

```bash
# 按解耦后的并行批次创建（批次1示例：3个并行任务）
git worktree add [DEV_DIR]/.worktree/[feature]-codex   -b feat/[feature]-codex
git worktree add [DEV_DIR]/.worktree/[feature]-sonnet  -b feat/[feature]-sonnet
git worktree add [DEV_DIR]/.worktree/[feature]-trae    -b feat/[feature]-trae

# 验证
git worktree list
```

**各 worktree 分配任务（来自 Phase 4.5 并行执行计划）：**

| Worktree | 工具 | 分配任务 | 特点 |
|----------|------|----------|------|
| worktree-feat-codex | Codex 插件版 | 批次1任务1 | 慢但稳定，上下文不丢失，适合复杂逻辑 |
| worktree-feat-sonnet | Antigravity Sonnet | 批次1任务2 | 速度快，适合中等复杂度 |
| worktree-feat-trae | Trae + gpt-5-codex | 批次1任务3 | 省 Token，适合简单任务 |

**每个 worktree 内的执行循环（严格按此顺序）：**

```
1. Codex MCP 执行任务（步骤 3.4）
2. Claude Code 验收门禁（步骤 3.4.5）— 必须通过才能继续
3. 写入文件，运行测试（步骤 3.5）
4. 全量测试无回归（步骤 3.6）
5. git commit（步骤 3.7）
6. 重复下一个任务
```

**Session 管理（并行时每个 worktree 独立）：**

```bash
# 每个 worktree 保存各自的 threadId
# worktree-1: threadId = "xxx-1"
# worktree-2: threadId = "xxx-2"
# worktree-3: threadId = "xxx-3"
# 各自用 codex_reply 追问，互不干扰
```

- [ ] worktree 已按并行批次创建
- [ ] 各 worktree 任务已分配
- [ ] 各 worktree 独立开发完成（含 CC 验收）

决策：

## 阶段三：Debug / 复杂问题阶段

### Step 8：复杂问题分析

遇到难以定位的 Bug 或架构问题时，双管齐下：

#### 方式 A：Codex 超高推理模式
```
[Codex 插件版，推理模式：超高]
问题描述：[详细描述现象、复现步骤、已尝试的方案]
相关代码：[粘贴关键代码片段]
请深度分析根本原因，给出修复方案。
```

#### 方式 B：Claude Opus Plan Mode
```
[Antigravity，Plan Mode + Opus]
这是一个复杂 Bug，我需要你帮我系统性分析：
1. 可能的根本原因（列举所有可能性）
2. 排查优先级
3. 每个假设的验证方法
4. 推荐的修复方案
[附上错误信息、相关代码、已尝试的方案]
```

**选择策略：**
- 代码逻辑问题 → Codex 超高推理
- 架构/设计问题 → Opus Plan Mode
- 不确定 → 两者都用，交叉验证

- [ ] 复杂问题已定位并修复

### Step 9：Opus 合并分支 + 冲突解决 + e2e 验证

**由 Opus 主导合并，原因：** 并行 worktree 可能修改了相同区域，冲突解决需要理解架构意图，Opus 最适合。

**9.1 合并前准备**

```bash
cd [PROJECT_ROOT]
git switch main
git pull --ff-only origin main

# 查看各分支与 main 的差异摘要
git log --oneline main..feat/[feature]-codex
git log --oneline main..feat/[feature]-sonnet
git log --oneline main..feat/[feature]-trae
```

**9.2 Opus 主导合并策略**

```
[Antigravity，Plan Mode + Opus]

我有以下并行开发的分支需要合并到 main：
- feat/[feature]-codex：负责任务 [列表]
- feat/[feature]-sonnet：负责任务 [列表]
- feat/[feature]-trae：负责任务 [列表]

开发计划：[STEPS_DIR]xxx-steps.md

请帮我：
1. 分析各分支的改动范围，预测可能的冲突点
2. 建议合并顺序（先合哪个分支冲突最少）
3. 对每个预期冲突点，给出解决策略
4. 如果各分支对同一文件有不同实现，判断哪个更优
```

**9.3 执行合并**

```bash
# 按 Opus 建议的顺序逐个合并
git merge --no-ff feat/[feature]-codex -m "feat: merge codex branch - [任务列表]"

# 如有冲突
git status   # 查看冲突文件
# 解决冲突后
git add [冲突文件]
git merge --continue

# 合并第二个分支
git merge --no-ff feat/[feature]-sonnet -m "feat: merge sonnet branch - [任务列表]"
# ... 依此类推
```

**冲突解决原则（Opus 判断）：**
- 同一功能的不同实现 → 选质量更高的，或手动整合两者优点
- 不同功能修改了同一文件 → 两者都保留，手动合并
- 一方删除、另一方修改 → 询问 Opus 判断意图

**9.4 e2e 测试门禁（合并后必须执行）**

```bash
# 单元测试全量
npm test   # 或 cmake --build build && ctest --output-on-failure

# e2e 测试（如有）— 这是合并到 main 前的最后一道门禁
npm run test:e2e
# 或 npx playwright test
# 预期：全部通过

# 如果 e2e 失败 → 不得推送 main，回到 Step 8 定位问题
```

**e2e 测试覆盖范围（至少包含）：**
- [ ] 本次新增功能的主路径
- [ ] 与新功能交互的现有功能（回归）
- [ ] 边界条件（空数据、最大值、异常输入）

- [ ] 合并顺序已由 Opus 确定
