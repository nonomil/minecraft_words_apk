# CLAUDE.md 审查报告

## 🔴 高优先级问题

### 1. 必填参数中缺少 `reasoning` 参数位置
**位置**：`Codex MCP 调用规范 → 必填参数`
**问题**：上一轮讨论确认可以加 reasoning，但文件里没有体现，会导致 CC 每次调用时遗漏。
**建议**：
```javascript
{
  model: "gpt-5.4",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  reasoning: "high" | "medium" | "low"  // 按任务类型选择，见 Skill 选择规则
}
```
并在 Skill 选择规则表中每行补充推荐的 reasoning 值。

---

### 2. Prompt 模板中 `reasoning` 指令与参数层重复但不一致
**位置**：`Codex Prompt 模板 → Constraints → 推理强度`
**问题**：Constraints 里的推理强度文字描述 和 参数层 `reasoning` 字段是两套机制，现在只有文字版，参数版缺失。两者如果不同步会产生歧义。
**建议**：保留 Prompt 层描述作为"备注说明"，明确注释"以参数层 reasoning 为准，此处仅作可读性说明"。

---

### 3. 模式升级触发条件与简单模式验证门禁位置混乱
**位置**：`简单模式流程`
**问题**：`⚠️ 模式升级触发条件` 夹在 Step 3 和 Step 4 之间，逻辑上应该是贯穿整个流程的**持续检查项**，而不是 Step 3 完成后才看一眼。容易被 CC 理解为"Step 3 之后再检查"。
**建议**：将其提到简单模式流程开头，标注为"全程持续检查"。

---

## 🟡 中优先级问题

### 4. 场景路由表与路由优先级顺序不一致
**位置**：`场景路由表` vs `路由优先级`
**问题**：路由表里顺序是"简单→复杂→Debug→并行→调研→大型代码库"，但优先级列表是"Debug→调研→大型→复杂→简单"，两者顺序相反，容易让 CC 按路由表顺序判断而忽略优先级。
**建议**：路由表按优先级从高到低排列，与优先级列表保持一致。

---

### 5. `claude-workflow-constants.md` 被多处引用但内容不在本文件
**位置**：`Codex MCP 调用规范` 开头
**问题**：三处关键规范（调用规范/文件边界/推理强度规则）都是"详见 constants.md"，但 CC 在主文件中无法直接获取这些约束，存在"规则在别处、执行在当前"的脱节风险。
**建议**：将最关键的不可变约束（如禁止命令清单、diff ≤ 200行）在本文件中内联一份摘要，constants.md 保留完整版。

---

### 6. 强制门禁 Step 3 中"读取 lessons.md"是括号备注，优先级不够高
**位置**：`强制门禁 Step 3`
**问题**：`读取 tasks/lessons.md` 写在括号里（"若存在"），给人一种"可选项"的感觉，但核心原则里说它是"强制执行"的。
**建议**：单独列为 Step 3.1，明确写"必须执行，文件不存在则跳过但需记录"。

---

### 7. 验证门禁的"Staff Engineer 三问"与"Linus 三问"功能重叠
**位置**：`简单模式 Step 3 验证门禁` vs `核心原则 → Linus 三问`
**问题**：两处都是代码质量自问，但分散在不同位置，内容相似却措辞不同，CC 可能只执行其中一处。
**建议**：合并为统一的"质量自检清单"，在简单模式和复杂模式中都引用同一份。

---

### 8. Self-Improvement Loop 中 lessons.md 写入时机不完整
**位置**：`核心原则 → 自我改进循环`
**问题**：目前只覆盖"用户纠正后"和"Bug修复后"两个时机，遗漏了：
- Review 超过 3 轮时的教训
- 模式升级触发时的教训（说明初始判断有误）
**建议**：补充这两个写入时机。

---

## 🟢 低优先级 / 体验优化

### 9. Codex Prompt 模板缺少"输出格式"约束
**问题**：没有要求 Codnet 输出结构化结果（如改动文件列表、验收自检结果），导致 CC 拿到结果后还需要二次解析。
**建议**：在模板末尾加：
```
## Output Format
完成后输出：
1. 改动文件列表（路径 + 改动行数）
2. 验收自检结果（逐条打勾）
3. 遗留问题（若有）
```

### 10. `并行开发` 路由触发条件过于简单
**问题**：`任务数 ≥ 2 且可解耦` 的判断标准没有说明如何判断"可解耦"，CC 可能随意判断。
**建议**：补充解耦判断标准，例如"改动文件集合无交集 + 无共享状态依赖"。

### 11. 项目特定信息区域缺少版本/更新时间戳
**问题**：`Codebase Overview` 和扫描摘要是静态内容，没有标注"截至时间"，时间长了会过期失效。
**建议**：扫描摘要已有时间戳（已做好），但 Codebase Overview 建议补充最后更新日期。
