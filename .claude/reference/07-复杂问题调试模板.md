# 06｜AI 调试阶段模板：复杂问题分析 → 定位 → 修复 → 验证

> 通用模板：适用于难以定位的 Bug、性能问题、架构缺陷的系统性分析。
> 工具组合：Codex 超高推理 + Claude Code Plan Mode 双管齐下。
> 将 `[BUG_*]` 占位符替换为实际信息后使用。

---

## 前置信息（填写后开始）

| 变量 | 说明 | 示例 |
|------|------|------|
| `[BUG_ID]` | Bug 标识 | `bug-2026-02-26-storage-crash` |
| `[BUG_TYPE]` | 问题类型 | `逻辑错误 / 性能 / 崩溃 / 数据丢失` |
| `[SYMPTOM]` | 现象描述 | `存档导出后导入失败` |
| `[PROJECT_ROOT]` | 项目根目录 | `G:/UserCode/XX/` |
| `[DEBUG_DIR]` | 调试记录目录 | `docs/debug/` |

---

## 完整流程概览

```
Phase 0   问题收集与分类
Phase 1   快速排查（5分钟内能定位则直接修复）
Phase 2   系统性分析（Codex 超高推理）
Phase 3   架构层分析（CC Plan Mode）
Phase 4   交叉验证（两者结论对比）
Phase 5   修复实施
Phase 6   回归验证
Phase 7   根因文档记录
```

---

## Phase 0：问题收集与分类

在开始分析前，填写问题卡片：

```markdown
## Bug 卡片：[BUG_ID]

**现象：** [用户/测试看到了什么]

**复现步骤：**
1. [步骤1]
2. [步骤2]
3. [步骤3]

**预期行为：** [应该发生什么]
**实际行为：** [实际发生了什么]

**错误信息：**
```
[粘贴完整错误堆栈 / 控制台输出]
```

**环境：**
- 浏览器/平台：
- 版本：
- 复现频率：必现 / 偶现（概率：__）

**已尝试的方案：**
- [ ] [方案1] → 结果：
- [ ] [方案2] → 结果：

**影响范围：** 单用户 / 所有用户 / 特定条件
**严重程度：** P0 崩溃 / P1 功能不可用 / P2 体验问题
```

**分类决策：**
```
复现频率 + 错误信息清晰？
  ├── 是 → Phase 1 快速排查（5分钟）
  └── 否 → 直接 Phase 2 系统性分析
```

- [ ] 问题卡片已填写
- [ ] 分类已确定

## Phase 1：快速排查（≤5 分钟）

先做最简单的检查，避免用大炮打蚊子：

```bash
# 1. 确认是否是最新代码的问题
git log --oneline -10

# 2. 确认测试是否覆盖了这个场景
npm test -- --grep "[相关测试名]"

# 3. 查看最近相关文件的改动
git log --oneline -5 -- [相关文件路径]
git diff HEAD~3 -- [相关文件路径]
```

**快速排查清单：**
- [ ] 是否是最近某次提交引入的？（`git bisect` 可定位）
- [ ] 是否有相关测试失败？
- [ ] 错误信息是否直接指向了问题代码？
- [ ] 是否是环境/配置问题（而非代码问题）？

**决策：**
- 5 分钟内定位 → 直接修复，跳到 Phase 5
- 未定位 → 进入 Phase 2

## Phase 2：系统性分析（Codex 超高推理）

**工具：** Codex 插件版，**推理模式：超高（o1/o3 级别）**

**输入 Prompt（完整版）：**

```
## Bug 分析请求

### 问题描述
现象：[SYMPTOM]
复现步骤：
1. [步骤1]
2. [步骤2]

错误信息：
```
[完整错误堆栈]
```

### 相关代码
文件：[文件路径]
```javascript
[粘贴相关代码，包含上下文，不要只粘贴出错行]
```

### 已尝试的方案
- [方案1] → 结果：[结果]
- [方案2] → 结果：[结果]

### 分析要求
请进行深度根因分析：

1. **假设列举**：列出所有可能导致此问题的原因（不要过早排除）
2. **概率排序**：按可能性从高到低排序
3. **验证方法**：每个假设如何用最小代价验证（加 console.log / 写单测）
4. **根因推断**：基于现有信息，最可能的根本原因是什么？
5. **修复方案**：针对根本原因的最小修复方案
6. **副作用分析**：修复方案可能影响哪些其他功能？

请不要给出"可能是X也可能是Y"的模糊答案，给出你最有把握的结论。
```

**记录 Codex 结论：**

```markdown
## Codex 分析结论（[日期]）

**最可能根因：** [结论]
**置信度：** 高/中/低
**验证方法：** [具体步骤]
**建议修复：** [方案]
**潜在副作用：** [列表]
```

- [ ] Codex 超高推理分析完成
- [ ] 结论已记录

## Phase 3：架构层分析（CC Plan Mode）

**工具：** Claude Code，**Shift+Tab 进入 Plan Mode**

适用场景：
- Codex 结论置信度低
- 问题涉及多个模块交互
- 怀疑是架构设计问题而非单点 Bug

**输入 Prompt：**

```
## 复杂 Bug 架构分析

这是一个难以定位的问题，需要从架构层面分析。

### 问题描述
[粘贴 Phase 0 的 Bug 卡片]

### Codex 的分析结论
[粘贴 Phase 2 的 Codex 结论]

### 请从架构角度分析：

1. **系统边界**：这个问题发生在哪个模块边界？
   - 数据流：[描述数据如何在模块间流动]
   - 状态管理：[哪些模块持有相关状态]

2. **根因假设**：
   - 是否是模块间接口设计问题？
   - 是否是状态同步问题？
   - 是否是时序/异步问题？
   - 是否是数据格式约定不一致？

3. **Codex 结论验证**：
   - Codex 认为根因是 [X]，你是否同意？
   - 如果不同意，你的根因假设是什么？

4. **修复策略**：
   - 最小修复（治标）：[方案]
   - 根本修复（治本）：[方案]
   - 两者的权衡：[分析]

5. **影响评估**：
   - 修复后需要回归测试哪些功能？
   - 是否需要数据迁移？
```

**记录 Opus 结论：**

```markdown
## Opus 分析结论（[日期]）

**根因判断：** [结论]
**与 Codex 是否一致：** 是/否
**分歧点：** [如有]
**建议修复策略：** 最小修复 / 根本修复
**回归测试范围：** [列表]
```

- [ ] Opus 架构分析完成
- [ ] 结论已记录

## Phase 4：交叉验证

对比 Codex 和 Opus 的结论：

```markdown
## 结论对比

| 维度 | Codex 结论 | Opus 结论 | 一致？ |
|------|-----------|-----------|--------|
| 根本原因 | | | |
| 修复方案 | | | |
| 影响范围 | | | |
| 修复优先级 | | | |
```

**决策规则：**
- 两者一致 → 直接进入 Phase 5 修复
- 两者不一致 → 执行验证实验（加日志/写单测），用数据决定
- 验证实验后仍不确定 → 采用"最小修复"策略，先止血再根治

**验证实验（如需）：**

```javascript
// 在关键路径加临时日志，验证假设
console.log('[DEBUG][BUG_ID] 关键变量:', {
  变量名: 变量值,
  时间戳: Date.now()
});
```

```bash
# 运行针对性测试
npm test -- --grep "[相关测试]"
```

- [ ] 结论对比完成
- [ ] 分歧已通过实验验证（如需）
- [ ] 最终修复方案已确定

## Phase 5：修复实施

**修复原则：**
- 最小改动：只改导致 Bug 的代码，不顺手重构
- 先写测试：修复前先写一个能复现 Bug 的测试
- 一次一个：不要同时修复多个 Bug

**步骤 5.1：写复现测试（先确认测试失败）**

```bash
# 写测试，描述期望行为
# 运行确认失败（复现 Bug）
npm test -- --grep "[Bug 复现测试名]"
# 预期：FAIL
```

**步骤 5.2：实施修复**

由 Codex 执行修复（参考《环境配置与协作规范》MCP 调用格式）：

```
mcp__codex__codex({
  model: "gpt-5-codex",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "
    ## Context
    - Bug: [BUG_ID] - [SYMPTOM]
    - Root cause: [Phase 4 确定的根因]
    - Files: [文件路径]: [用途]

    ## Task
    修复 [BUG_ID]：[一句话描述修复内容]
    Steps:
    1. [具体修复步骤]
    2. [移除临时调试日志]

    ## Constraints
    - Scope: Only modify [指定文件]
    - API: Don't change existing function signatures
    - Deps: No new dependencies
    - Style: Follow existing code style

    ## Acceptance
    - [ ] Bug 复现测试通过
    - [ ] 全量测试无回归
  "
})
```

**步骤 5.3：验证修复**

```bash
# 确认 Bug 复现测试通过
npm test -- --grep "[Bug 复现测试名]"
# 预期：PASS

# 全量测试无回归
npm test
# 预期：全部 PASS
```

**步骤 5.4：移除调试日志并提交**

```bash
# 确认没有遗留 console.log('[DEBUG]...')
grep -r "\[DEBUG\]\[" src/

git add [修改的文件]
git commit -m "fix: [BUG_ID] - [一句话描述修复]"
```

- [ ] 复现测试已写并确认失败
- [ ] 修复已实施
- [ ] 复现测试通过
- [ ] 全量测试无回归
- [ ] 调试日志已清理
- [ ] 已提交

## 快速参考：工具选择

| 问题类型 | 首选工具 | 备选 |
|----------|----------|------|
| 代码逻辑错误 | Codex 超高推理 | Opus 辅助 |
| 异步/时序问题 | Codex 超高推理 | 加日志验证 |
| 模块间接口问题 | CC Plan Mode | Codex 验证 |
| 架构设计缺陷 | CC Plan Mode | — |
| 性能问题 | Codex（分析热点） | Chrome DevTools |
| 数据丢失/损坏 | Opus（数据流分析） | Codex（代码层） |
| 偶现问题 | 加日志 → Codex 分析日志 | — |

## 注意事项

1. **不要跳过 Phase 0**：没有完整的问题描述，任何分析都是猜测
2. **调试日志必须清理**：提交前用 `grep -r "\[DEBUG\]"` 确认
3. **先写测试再修复**：没有复现测试的修复可能只是掩盖了问题
4. **最小修复原则**：Bug 修复不是重构机会，保持改动最小
5. **记录分析过程**：Phase 7 的文档是团队知识资产，不要省略
