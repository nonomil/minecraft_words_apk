## 模式对比总览

| 特性 | 临时分支模式 | 固定分支模式（Bare Repo） |
|------|-------------|------------------------|
| **适用场景** | 短期功能开发、Bug 修复 | 长期维护版本、多版本并行 |
| **分支生命周期** | 临时，合并后删除 | 长期存在，持续维护 |
| **worktree 位置** | `worktrees/` 或 `.worktree/` 子目录 | 项目根目录平铺展示 |
| **git 管理** | 统一在根目录 `.git` | `.bare/` 集中管理所有 git 数据 |
| **视觉清爽度** | ★★★☆☆ | ★★★★★ |
| **复杂度** | 低 | 中 |
| **初始化方式** | `claude-workflow-init.md` Step 3 选 1 | `claude-workflow-init.md` Step 3 选 2 |

## 模式一：临时分支模式

### 特点

- 分支都是临时的，开发完合并入主分支后可删除
- 不维护分支的独立 git，所有操作通过根目录的 git 进行
- worktree 创建在子目录中（如 `worktrees/feat-xxx`）

### 目录结构

```
project-root/                 ← 主分支（master/main），包含 .git/
├── .git/                     ← 唯一的 git 仓库
├── src/                      ← 主分支代码
├── docs/
├── worktrees/                ← worktree 目录（非隐藏，推荐）
│   ├── feat-login/           ← feature/login 分支工作区
│   │   ├── src/              ← 与主分支隔离的工作副本
│   │   └── ...
│   └── fix-bug-123/          ← bugfix 分支工作区
└── ...
```

### 临时分支工作流程

```
┌─────────────────────────────────────────────────────────────┐
│  1. 创建功能分支 worktree                                    │
│     git worktree add worktrees/feat-xxx -b feature/xxx       │
├─────────────────────────────────────────────────────────────┤
│  2. 在隔离环境中开发                                         │
│     cd worktrees/feat-xxx                                    │
│     # 编码 → 测试 → commit                                   │
├─────────────────────────────────────────────────────────────┤
│  3. 保持与主分支同步（可选）                                 │
│     git fetch origin                                         │
│     git rebase origin/main                                   │
├─────────────────────────────────────────────────────────────┤
│  4. 合并回主分支                                             │
│     cd ../..                                                 │
│     git merge feature/xxx                                    │
├─────────────────────────────────────────────────────────────┤
│  5. 清理                                                     │
│     git worktree remove worktrees/feat-xxx                   │
│     git branch -d feature/xxx                                │
└─────────────────────────────────────────────────────────────┘
```

### 临时分支常用命令

#### 创建 worktree

```bash
# 基于当前 HEAD 创建新分支和 worktree
git worktree add worktrees/feat-login -b feature/login

# 基于指定分支创建 worktree
git worktree add worktrees/fix-bug -b fix/bug-123 origin/main

# 基于 tag 创建 worktree（用于 hotfix）
git worktree add worktrees/hotfix-v1.2 v1.2.0
```

#### 日常开发

```bash
# 进入 worktree 开发
cd worktrees/feat-login

# 正常 git 操作（与常规仓库相同）
git add .
git commit -m "feat: login"
git push -u origin feature/login

# 查看当前在哪个 worktree
pwd
# /path/to/project/worktrees/feat-login
```

#### 同步与更新

```bash
# 方法1：在 worktree 内同步主分支更新
cd worktrees/feat-login
git fetch origin
git rebase origin/main

# 方法2：在主仓库更新所有 worktree
cd ../..
git fetch origin
cd worktrees/feat-login
git rebase origin/main
```

#### 合并与清理

```bash
# 回到主仓库
cd ../..

# 合并功能分支
git merge --no-ff feature/login -m "feat: merge login feature"

# 推送主分支
git push origin main

# 清理 worktree（先确保没有未提交改动）
git worktree list
# 检查状态干净后
git worktree remove worktrees/feat-login

# 删除远程分支（如已合并）
git push origin --delete feature/login

# 删除本地分支引用
git branch -d feature/login
```

### 临时分支清理策略

| 场景 | 命令 |
|------|------|
| 删除单个 worktree | `git worktree remove worktrees/feat-xxx` |
| 强制删除（有未提交改动） | `git worktree remove -f worktrees/feat-xxx` |
| 只移除记录（保留目录） | `git worktree prune` |
| 批量清理已完成分支 | 脚本遍历 `git worktree list` 检查 merged 状态 |

## 模式二：固定分支模式（Bare Repo）

### 特点

- 每个分支完全独立，像独立项目一样存在
- 适合需要长期维护的版本（如 v1.x, v2.x）
- **视觉上最清爽**：worktree 平铺在项目根目录，没有嵌套感
- 使用 `.bare/` 文件夹集中管理所有 git 数据

### 目录结构

```
amend-crash-demo/             ← 项目根目录
├── .bare/                    ← bare repo，存储所有 git 数据（隐藏）
│   ├── HEAD
│   ├── config
│   ├── objects/
│   └── refs/
├── .git                      ← 文件，内容为 "gitdir: ./.bare"
├── main/                     ← main 分支工作区
│   ├── src/
│   └── README.md
├── v1.x/                     ← v1.x 维护分支（长期存在）
│   ├── src/
│   └── README.md
├── v2.x/                     ← v2.x 维护分支（长期存在）
│   ├── src/
│   └── README.md
└── feature3/                 ← feature 分支（短期）
    ├── src/
    └── feature3.yaml
```

**`ls` 看到的只有干净的文件夹：**

```bash
$ ls
main/    v1.x/    v2.x/    feature3/

$ ls -a
.bare/   .git     main/    v1.x/    v2.x/    feature3/
```

### Bare Repo 初始化流程

#### 方式 A：新项目初始化（推荐）

```bash
# 1. 创建项目目录
mkdir amend-crash-demo
cd amend-crash-demo

# 2. 以 bare 形式 clone
git clone --bare git@github.com:user/repo.git .bare

# 3. 创建 .git 文件指向 .bare
echo "gitdir: ./.bare" > .git

# 4. 修改 .bare/config，确保 fetch 配置正确
cat >> .bare/config << 'EOF'
[remote "origin"]
    url = git@github.com:user/repo.git
    fetch = +refs/heads/*:refs/remotes/origin/*
EOF

# 5. 创建 main worktree
git worktree add main

# 6. 进入 main 开始开发
cd main
```

#### 方式 B：现有项目转换为 Bare Repo

```bash
# 1. 备份当前 .git
mv .git .git-backup

# 2. 转换为 bare repo
git clone --bare .git-backup .bare

# 3. 创建 .git 文件
echo "gitdir: ./.bare" > .git

# 4. 配置 fetch
vim .bare/config
# 添加：fetch = +refs/heads/*:refs/remotes/origin/*

# 5. 创建 main worktree
git worktree add main

# 6. 移动原项目代码到 main worktree
mv *.py *.md src/ tests/ docs/ main/ 2>/dev/null || true

# 7. 提交整理
cd main
git add -A
git commit -m "chore: 转换为 Bare Repo 格式"

# 8. 清理备份
cd ..
rm -rf .git-backup
```

### Bare Repo 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│  项目根目录（只有 .bare/ 和 .git，无实际代码）               │
│  ├── .bare/        ← 所有 git 数据                          │
│  └── .git          ← gitdir: ./.bare                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │   main/     │  │    v1.x/    │  │  feature3/  │         │
│  │             │  │             │  │             │         │
│  │ • src/      │  │ • src/      │  │ • src/      │         │
│  │ • tests/    │  │ • tests/    │  │ • tests/    │         │
│  │ • README    │  │ • README    │  │ • feature3  │         │
│  │             │  │             │  │   .yaml     │         │
│  │ [生产环境]   │  │ [维护分支]   │  │ [开发中]    │         │
│  └─────────────┘  └─────────────┘  └─────────────┘         │
│                                                             │
│  各分支完全隔离，可同时 checkout 不同分支                   │
│  没有 .git 目录干扰，视觉清爽                               │
└─────────────────────────────────────────────────────────────┘
```

### Bare Repo 常用命令

#### 查看状态

```bash
# 在项目根目录执行（因为有 .git 文件）
git worktree list
# /path/to/amend-crash-demo/.bare        (bare)
# /path/to/amend-crash-demo/feature3     aeaac94 [feature/JIRA234-feature3]
# /path/to/amend-crash-demo/main         82b8711 [main]
# /path/to/amend-crash-demo/v1.x         xyz7890 [v1.x]

# 查看所有分支（包括远程）
git branch -a
```

#### 创建新分支 worktree

```bash
# 基于 main 创建 feature 分支
git worktree add -b "feature/new-feature" new-feature main

# 基于 v1.x 创建 hotfix 分支
git worktree add -b "hotfix/critical-fix" hotfix-v1 v1.x

# 基于远程分支
git worktree add --track -b feature/xxx feature-xxx origin/feature/xxx
```

#### 在各分支独立开发

```bash
# 进入 feature3 开发
cd feature3
echo "feature3 development" > feature3.yaml
git add .
git commit -m "feat: feature3"
git push --set-upstream origin feature/JIRA234-feature3

# 切换到 main 做 hotfix
cd ../main
echo "hotfix" > hotfix.yaml
git add .
git commit -m "fix: critical bug"
git push

# 把 hotfix 同步到 v1.x
cd ../v1.x
git cherry-pick main~1  # 或手动合并
git push
```

#### 清理已完成的 worktree

```bash
# 回到项目根目录
cd ..

# 移除 worktree（保留分支历史）
git worktree remove feature3

# 如果目录非空，强制移除
git worktree remove -f feature3

# 清理无效记录
git worktree prune

# 删除远程分支（如已合并）
git push origin --delete feature/JIRA234-feature3
```

### 多版本并行管理

```
┌─────────────────────────────────────────────────────────────┐
│                    Bare Repo 多版本并行                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   main/          v1.x/           v2.x/         feature/      │
│    ↓              ↓                ↓              ↓         │
│  最新开发       稳定版1.x        稳定版2.x      新功能开发    │
│  (不稳定)      (仅bugfix)       (仅bugfix)     (实验性)      │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  典型操作流：                                                │
│                                                             │
│  1. 在 feature/ 开发新功能                                   │
│  2. 完成后合并到 main                                        │
│  3. main 稳定后打 tag，创建新的稳定版 worktree              │
│  4. 继续下一轮开发                                           │
│                                                             │
│  5. 发现 v1.x 的 bug：                                       │
│     - 在 v1.x/ 修复                                          │
│     - cherry-pick 到 main                                    │
│     - 如有 v2.x，评估是否需要同步                            │
└─────────────────────────────────────────────────────────────┘
```

## 如何选择

### 决策树

```
分支需要长期存在？
├── 是 → 固定分支模式（Bare Repo）
│       └── 示例：v1.x, v2.x, LTS 版本维护
│
└── 否 → 开发周期多长？
        ├── < 1 周 → 临时分支模式（简单）
        └── > 1 周 → 固定分支模式（体验更好）
```

### 快速对照表

| 你的情况 | 推荐模式 |
|---------|---------|
| 单人项目，偶尔开发 | 临时分支 |
| 团队协作，频繁切换任务 | 临时分支 |
| 需要同时维护多个发布版本 | **Bare Repo** |
| 对目录整洁度有强迫症 | **Bare Repo** |
| 大型重构，需要长期隔离 | **Bare Repo** |
| 只是想试试 worktree | 临时分支 |

## 故障排除

### 临时分支模式问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `worktree is already locked` | 之前的操作中断 | `git worktree unlock worktrees/xxx` |
| `previously used worktrees/xxx` | 残留记录 | `git worktree prune` |
| 权限错误 | 目录被占用 | 关闭编辑器/IDE 后重试 |

### Bare Repo 模式问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `not a git repository` | 不在正确目录 | 确保在项目根目录（有 .git 文件的目录） |
| `gitdir path does not exist` | .bare 被移动 | 修复 .git 文件中的路径 |
| 分支名显示不正确 | fetch 配置缺失 | 编辑 .bare/config 添加 `fetch = +refs/heads/*:refs/remotes/origin/*` |
| push 失败 | 没有设置 upstream | `git push -u origin branch-name` |

### 通用问题

```bash
# 查看所有 worktree 详情
git worktree list --porcelain

# 修复损坏的 worktree 记录
git worktree repair

# 强制清理所有无效记录
git worktree prune -v
```
