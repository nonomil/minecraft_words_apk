# AI开发工作流 · 快速上手指南

> 5分钟读完，掌握日常开发80%的场景。
> 遇到复杂情况再查原始文档（docs/目录）。

---

## 第一步：判断用哪个模式

```
改动预估 ≤ 200行 且 涉及 ≤ 3个文件 且 需求很清楚？
  → YES：简单模式（10-30分钟搞定）
  → NO ：复杂模式（需要先做计划）
```

不确定就先按简单模式启动，遇到以下情况立刻切换复杂模式：
- 发现要改超过3个文件
- diff行数明显要超200行
- 需求在开发中变大了
- Review超过3轮还有新问题

## 简单模式：4步走

**Step 1 — 让AI先分析，别急着动代码**

在 Claude Code 里按 `Shift+Tab` 进入 Plan Mode，输入：
```
分析 [你的任务]，先不要改代码。
列出要改哪些文件，改什么，回答：
1. 这是真实问题还是想象的？
2. 有没有更简单的方式？
3. 会破坏哪些现有功能？
等我确认后再动手。
```
看完AI的分析，觉得没问题，回复"确认"。

**Step 2 — 执行**

- AI会调用 Codex 生成代码，或者直接改（20行以下的小改动）
- 你不需要操作，等它完成就行

**Step 3 — Review（必须，不能跳过）**

AI完成后，让它做两轮独立Review：
```
请先做轮次1：你自己审查刚才的改动，检查逻辑、安全问题、有没有改多余的东西。
完成后，再做轮次2：用Codex独立审查边界条件和长尾情况。
两轮都通过再告诉我结果。
```
如果发现问题，让AI修复后重新Review，直到两轮都无新问题。

**Step 4 — 提交**

```
git diff --stat   # 确认行数 ≤ 200
git add [文件]
git commit -m "feat/fix: [描述]"
```

## 复杂模式：分5个阶段

### 阶段1：出计划（30-60分钟）

打开 Antigravity，切换到 **Opus 模型 + Plan Mode**，把需求丢进去：
```
我需要实现 [需求描述]。
帮我澄清歧义、识别风险、输出方案文档，保存到 docs/plan/YYYY-MM-DD-[名字].md
```
看完文档，回答AI提出的待确认问题。

### 阶段2：交叉Review计划（20-40分钟）

让 Codex 从工程角度审查这份计划：
```
请审查 docs/plan/xxx.md，从工程实现角度找问题：可行性、遗漏细节、依赖顺序、边界条件。
```
再让 Opus 回应 Codex 的意见，反复2-3轮直到没有新问题。

### 阶段3：拆分任务（关键，不能省）

让 Codex 生成开发步骤：
```
基于 docs/plan/xxx.md，生成step-by-step开发计划。
每个任务diff ≤ 200行，有独立测试验证，保存到 docs/development/xxx-steps.md
```

**你要人工检查**：每个任务是否独立、文件是否有冲突、依赖顺序是否合理。不过这关才能开始写代码。

### 阶段4：开发

每个任务按简单模式的Step 1-4执行，完成一个提交一个，不要攒着。

复杂任务可以多开几个worktree并行：
```bash
git worktree add [DEV_DIR]/.worktree/feat -b feat/xxx
# 在不同目录里同时跑不同任务
```

### 阶段5：合并

```bash
git switch main
git pull --ff-only
git merge --no-ff feat/xxx -m "feat: merge xxx"
npm test          # 必须全过
git push origin main
```

## 日常使用的几条原则

**单次改动不超200行。** 超了就拆。这是防止Review地狱的最重要原则（超大改动可能需要Review 9小时）。

**Review不能跳过。** AI首轮代码大约只有60-70分，不Review直接上线是在埋雷。

**发现AI输出质量变差，立刻重启Session。** 尤其是长时间工作后或晚上，重新开一个会话把背景粘贴进去。

**计划和代码分开。** 复杂需求先出计划再写代码，不要边想边写。

## 常用命令速查

```bash
# 查看当前改动量
git diff --stat

# 创建并行工作区
git worktree add [DEV_DIR]/.worktree/xxx -b feat/xxx
git worktree list
git worktree remove [DEV_DIR]/.worktree/xxx

# 运行测试
npm test
npm test -- --grep "测试名称"   # 只跑特定测试

# 提交规范
git commit -m "feat: 新功能描述"
git commit -m "fix: 修复问题描述"
git commit -m "chore: 配置/文档更新"
```

## 遇到麻烦时

| 情况 | 怎么办 |
|------|--------|
| AI改了不该改的文件 | `git checkout -- [文件]` 回滚，重新给AI更明确的范围限制 |
| Review一直有新问题 | 任务太大，回阶段3重新拆分 |
| AI输出质量很差 | 重启Claude Code Session，重新提供背景 |
| 不知道该用简单还是复杂模式 | 先简单模式，触发升级条件再切换 |
| 合并冲突不知道怎么选 | 让Opus分析两个分支的意图，它来决策 |
