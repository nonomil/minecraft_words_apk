# AI开发工作流规范 — Claude Code 自动读取

> 本文件与项目原有 CLAUDE.md 并存，补充开发流程约束。
> 优先级：本文件规则 > 默认行为，但不覆盖原 CLAUDE.md 中的项目特定配置。

---

## 角色分工（铁律）

| 工具 | 职责 | 禁止 |
|------|------|------|
| **Claude Code (CC)** | 规划、搜索、决策、Review、Git操作、文档 | 默认不写业务代码（仅 <20 行且无逻辑改动例外） |
| **Codex MCP** | 代码生成、重构、Bug修复、测试代码 | 架构决策、Git操作 |

Codex调用三个必填参数，缺一不可：
`model: "gpt-5-codex"` · `sandbox: "danger-full-access"` · `approval-policy: "on-failure"`

---

## 任务启动决策树

```
收到任务
  ↓
diff预估 ≤ 200行 且 ≤ 3文件 且 需求明确？
  ├── YES → 简单模式
  └── NO  → 复杂模式

简单模式升级触发（立即暂停切换）：
  diff超200行 / 跨3+文件 / 需求边界扩大 / Review超3轮仍有新问题
```

---

## 简单模式

```
1. Plan Mode（Shift+Tab）
   → 回答Linus三问（见下方）
   → 列出改动文件，等用户确认

2. 实现
   → <20行无逻辑：CC直接改
   → 有逻辑：Codex MCP执行，保存threadId

3. Review 2轮（独立进行，不互相参考）
   → 轮次1：CC审查（逻辑/安全/YAGNI/向后兼容）
   → 轮次2：Codex深度审查（边界条件/长尾）
   → 连续2轮无新问题 → 进下一步
   → 有问题 → 修复后重新轮次1（上限3轮，超过→切复杂模式）

4. git diff --stat 确认 ≤ 200行 → commit
```

---

## 复杂模式

```
Phase 1  需求澄清
         Antigravity + Plan Mode + Opus
         输出：需求理解/技术方案/风险点/待确认问题
         保存：docs/plan/YYYY-MM-DD-[feature].md

Phase 2  Plan交叉Review（≥3轮，上限5轮）
         Codex工程审查 ↔ Opus架构回应
         分歧记入Plan文档"已知权衡"表
         连续2轮无新问题 → 定稿

Phase 3  任务解耦（强制门禁，不过不得开发）
         每个任务：diff ≤ 200行 / 独立验收 / 文件不重叠 / 依赖无环
         人工复核通过后继续

Phase 4  并行开发
         git worktree add [DEV_DIR]/.worktree/[branch] -b [branch]
         每个任务：Codex执行 → 写失败测试 → 确认FAIL → 实现 → 确认PASS → 多轮Review → commit
         Context刷新：单Session超3个任务 → 开新Session
         止损：Review超5轮/3h/50k token → 停止，回Phase 3重新拆分

Phase 5  合并
         Opus分析冲突 → 建议合并顺序 → merge → 全量测试 → e2e测试 → 推送
```

## Codex MCP调用模板

```javascript
mcp__codex__codex({
  model: "gpt-5.4",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: `
    ## Context
    Tech Stack: [语言/框架/版本]
    Files: [路径]: [用途]
    Reference: [风格参考文件]

    ## Task
    [单一、明确、可验证的任务描述]
    Steps: 1.[步骤1] 2.[步骤2]

    ## Constraints
    - API: 不改变 [已有函数签名]
    - Scope: 只修改 [指定文件]
    - Deps: 不新增依赖

    ## Acceptance
    - [ ] 测试通过
    - [ ] 不破坏现有功能
  `
})
// 追问：mcp__codex__codex_reply({ threadId: "上一步ID", ... })
```

## Linus三问（每次Plan Mode前必答）

1. 这是真实问题还是想象的？（拒绝过度工程）
2. 有没有更简单的方式？（永远找最简解）
3. 这会破坏什么现有功能？（向后兼容是铁律）

---

## 多轮Review协议

> AI首轮代码质量约60-70分，每轮只能召回部分问题，必须多轮。

```
轮次1  CC快速审查：逻辑/安全/YAGNI/向后兼容
轮次2  Codex深度审查：边界条件/长尾/测试覆盖（独立，不参考轮次1）
       CC结合需求context筛选确认真实问题
       Codex修复
轮次3  验证修复，确认无新问题
终止：连续2轮无新问题
上限：5轮仍有问题 → 任务粒度太大，拆分重做
```

| diff规模 | 最少Review轮次 |
|---------|--------------|
| ≤ 50行 | 2轮 |
| 50–200行 | 3轮 |
| > 200行 | 禁止，先拆分 |

## Git安全约束

未经用户明确确认，禁止执行：
`git stash` · `merge main` · `push main` · `reset --hard` · `checkout -- .` · `clean -f`

合并前必须输出：变更文件列表 + Commit列表 + CI状态，等用户回复"确认合并"再执行。

---

## 禁止行为

- CC在规划阶段写业务代码
- Codex调用缺少三个必填参数
- 跳过Linus三问
- Codex输出未经Review直接commit
- 单次变更超200行
- Context压缩后不刷新Session继续低质量输出
