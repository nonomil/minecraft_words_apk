# 04｜AI 计划阶段模板：需求分析 → Plan 生成 → 交叉 Review

> 通用模板：适用于任何复杂功能开发前的计划阶段。
> 工具组合：Claude Opus (Plan Mode) + Codex (gpt-5-codex) 交叉 Review。
> 将 `[FEATURE_*]` 占位符替换为实际信息后使用。

---

## 为什么 Antigravity/Kiro 的 Plan Mode 质量更高？

基于调研结论（来源：Anthropic 官方 + 社区实践）：

**1. Opus 模型本身的优势**
- 更强的长程连贯性：能维持复杂多步计划而不丢失上下文（1M token 窗口）
- 自我纠错能力：会主动发现自己推理中的矛盾并修正
- SWE-bench 74.5%：真实代码库任务表现最强，架构理解更深
- 专为 agentic 工作流优化：规划 → 工具调用 → 验收的完整链路

**2. Plan Mode 的机制优势**
- **只读探索**：Plan Mode 下 AI 只能读文件、不能改文件，强制先理解再行动
- **防止意外**：在任何代码被修改前，先展示完整计划供人审查
- **范围纪律**：Plan Mode 天然限制了 scope creep，每次计划控制在可执行范围内

**3. 实践结论**
- 最初的 Plan（需求澄清 + 架构决策）：用 **Antigravity + Opus + Plan Mode**，质量最高
- 后续开发中的小计划（单任务实现方案）：用 **Claude Code for VS Code + Plan Mode** 即可
- Codex 不做架构决策，只做工程审查和代码生成

## 高质量 Plan Prompt 模板（基于调研）

以下 Prompt 结合了 CoT（链式推理）、ToT（树状探索）、ADR 格式，经过验证能显著提升 Plan 质量：

### 模板 A：需求澄清（首次 Plan，用 Opus）

```
## 角色
你是一位资深软件架构师，擅长发现需求歧义和技术风险。

## 背景
项目：[项目名称和简介]
技术栈：[TECH_STACK]
相关文件：[列出关键文件路径]

## 需求
[详细描述需求，包括用户场景、期望行为、边界条件]

## 请按以下步骤思考（Chain-of-Thought）

**步骤 1：重述需求**
用你自己的话重述这个需求，确认我们理解一致。

**步骤 2：发现歧义**
列出所有你不确定的地方，每条歧义说明：
- 歧义描述
- 可能的解释 A
- 可能的解释 B
- 你倾向于哪种，理由是什么

**步骤 3：探索方案（Tree-of-Thought）**
列出 2-3 个可行的技术方案，每个方案说明：
- 核心思路
- 优点
- 缺点
- 适用条件

**步骤 4：推荐方案**
基于当前项目约束，推荐哪个方案？理由是什么？
明确说明你排除其他方案的原因。

**步骤 5：风险识别**
推荐方案的主要风险是什么？如何缓解？

**步骤 6：输出 Plan 文档**
将以上分析整理为 Plan 文档，保存到：
[PLAN_DIR]YYYY-MM-DD-[FEATURE_NAME].md

## 成功标准
- 所有歧义已明确
- 方案选择有充分理由
- 风险已识别并有缓解方案
- 另一个工程师读完 Plan 能独立实现
```

### 模板 B：架构决策（ADR 格式，用 Opus）

```
## 架构决策请求

**决策问题：** [一句话描述需要做的架构决策]

**背景约束：**
- 现有架构：[描述]
- 性能要求：[描述]
- 团队约束：[描述]
- 不可改变的：[列出硬约束]

**请输出 ADR（Architecture Decision Record）格式：**

### 状态
[提议中]

### 背景
[为什么需要做这个决策]

### 决策选项
| 选项 | 优点 | 缺点 | 复杂度 |
|------|------|------|--------|
| 选项A | | | |
| 选项B | | | |

### 决策
选择 [选项X]，理由：
1. [理由1]
2. [理由2]

### 后果
- 正面影响：[列表]
- 负面影响/权衡：[列表]
- 需要注意：[列表]
```

### 模板 C：Claude Code for VS Code Plan Mode（日常开发）

```
Shift+Tab 进入 Plan Mode，输入：

分析 [具体任务]，在修改任何代码前：
1. 搜索相关文件（Glob/Grep）
2. 回答 Linus 三问：
   - 这是真实问题还是想象的？
   - 有没有更简单的方式？
   - 这会破坏什么现有功能？
3. 列出将要修改的文件和改动点
4. 等我确认后再执行
```

## 自动化 vs 手动控制

| 步骤 | 控制方式 | 理由 |
|------|----------|------|
| Opus 需求澄清 | **手动** | 需要人确认 Opus 理解是否正确，歧义需人判断 |
| Codex 工程审查 | **手动** | 需要人决定哪些意见值得采纳 |
| 交叉 Review 轮次 | **手动** | 终止条件是主观判断，不可自动化 |
| Codex 生成 step-by-step | **可自动化** | 输入确定（定稿 Plan），输出格式固定 |
| Opus 审查开发计划 | **手动（可选）** | 质量门，需人判断是否通过 |

**核心原则：** 计划阶段的价值在于人工判断，自动化只适用于格式化输出步骤。

## 完整流程概览

```
Phase -1  复杂度判断 → 选择开发模式（简单 / 复杂）
Phase 0   资料调研（可选，见《AI资料调研搜索阶段模板》）
Phase 1   Opus 需求澄清 → 生成 Plan 文档        [复杂模式]
Phase 2   Codex 工程审查 Plan                   [复杂模式]
Phase 3   交叉 Review（Opus ↔ Codex，≥3轮）     [复杂模式]
Phase 4   Codex 生成 Step-by-Step 开发计划      [复杂模式]
Phase 4.5 任务解耦审查（三轮确认）               [复杂模式]
Phase 5   Opus 审查开发计划（可选）              [复杂模式]
Phase 6   计划定稿，进入开发阶段
```

## Phase 1：Opus 需求澄清 → 生成 Plan 文档

**工具：** Antigravity / Kiro，切换到 **Plan Mode + Opus 模型**

**输入 Prompt：**

```
我需要实现以下功能：
[详细描述需求，包括：用户场景、期望行为、边界条件、已知约束]

项目信息：
- 技术栈：[TECH_STACK]
- 项目根目录：[PROJECT_ROOT]
- 相关文件：[列出可能涉及的文件]

请帮我：
1. 重述你对需求的理解（确认我们在说同一件事）
2. 列出所有歧义点和待确认问题
3. 识别技术风险和依赖
4. 输出一份 Plan 方案文档，保存到 [PLAN_DIR]YYYY-MM-DD-[FEATURE_NAME].md
```

**Plan 文档标准结构：**

```markdown
# [FEATURE_NAME] Plan 方案

> 生成日期：YYYY-MM-DD
> 状态：草稿 / 审查中 / 定稿

## 流程进度
- [ ] Phase 1：Plan 文档生成，用户确认内容
- [ ] Phase 2：Codex 工程审查完成，用户确认采纳意见
- [ ] Phase 3：交叉 Review 收敛（连续2轮无新问题），用户确认定稿
- [ ] Phase 4：各 worktree 独立 steps 文档生成，用户确认
- [ ] Phase 4.5：解耦审查三轮通过，用户确认
- [ ] Phase 6：用户说"开始开发"，进入执行阶段

## 需求理解
[Opus 对需求的完整理解，含边界条件和排除项]

## 技术方案
### 方案选择
[列出 2-3 个可选方案，说明选择理由]

### 选定方案详述
[架构图（文字描述）、关键设计决策、数据流]

### 涉及文件
| 文件 | 操作 | 说明 |
|------|------|------|
| `src/xxx.js` | 修改 | [用途] |
| `src/yyy.js` | 新建 | [用途] |

## Worktree 并行计划
| Worktree | 分支 | 任务 | 依赖 | Steps 文档 |
|----------|------|------|------|-----------|
| .worktree/task-1 | feat/xxx-task-1 | [任务描述] | 无 | docs/development/xxx-task-1-steps.md |
| .worktree/task-2 | feat/xxx-task-2 | [任务描述] | task-1 接口 | docs/development/xxx-task-2-steps.md |

## 风险点
| 风险 | 概率 | 影响 | 缓解方案 |
|------|------|------|----------|
| [风险描述] | 高/中/低 | 高/中/低 | [方案] |

## 待确认问题
- [ ] [问题1]
- [ ] [问题2]

## 验收标准
- [ ] [可验证的标准1]
- [ ] [可验证的标准2]

## 已知权衡（Opus vs Codex 分歧）
| 分歧点 | Opus 观点 | Codex 观点 | 决策 | 理由 |
|--------|-----------|------------|------|------|
```

**完成检查：**
- [ ] Plan 文档已生成
- [ ] 文档已移入 `[PLAN_DIR]`
- [ ] 所有待确认问题已回答

## Phase 2：Codex 工程审查 Plan

**工具：** Codex 插件版（gpt-5-codex）

**输入 Prompt：**

```
请从工程实现角度审查这份 Plan 文档：
文件路径：[PLAN_DIR]YYYY-MM-DD-[FEATURE_NAME].md

审查维度：
1. 方案可行性：技术方案是否可以实现？有无技术障碍？
2. 遗漏细节：有哪些实现细节 Plan 中没有提到？
3. 步骤合理性：实现顺序是否正确？有无依赖关系问题？
4. 测试策略：如何验证每个步骤的正确性？
5. 边界条件：有哪些边界情况没有处理？
6. 性能/安全：有无明显的性能瓶颈或安全问题？

请输出：
- 问题列表（按优先级排序）
- 建议修改点
- 你认为可以直接开始实现的部分
```

**完成检查：**
- [ ] Codex 审查意见已收集
- [ ] Plan 文档已根据意见修改

## Phase 3：交叉 Review（2-3 轮）

**规则：**
- 每轮：Opus 分析 → Codex 审查 → 记录分歧
- 终止条件：双方无新问题，或分歧已明确记录为"已知权衡"
- 超过 3 轮仍有重大分歧 → 停止，重新澄清需求

### 轮次 1

**Opus Prompt：**
```
请重新审查 Plan 文档 [PLAN_DIR]xxx.md。
Codex 提出了以下工程问题：
[粘贴 Codex 的问题列表]

请逐条回应：
1. 你是否同意这个问题？
2. 如果同意，建议如何修改 Plan？
3. 如果不同意，理由是什么？
4. 是否有 Codex 遗漏的架构层面问题？
```

**Codex Prompt（轮次 1 后）：**
```
Plan 文档已根据上轮 Review 更新。
请再次审查 [PLAN_DIR]xxx.md：
1. 上轮提出的问题是否已解决？
2. 修改是否引入了新问题？
3. 现在是否可以开始生成开发计划？
```

### 轮次 2 / 3（按需重复）

重复上述 Prompt 模式，直到满足终止条件。

**分歧记录表（填入 Plan 文档末尾）：**

```markdown
## 已知权衡（Opus vs Codex 分歧）
| 分歧点 | Opus 观点 | Codex 观点 | 决策 | 理由 |
|--------|-----------|------------|------|------|
| [分歧] | [观点] | [观点] | 采用Opus/Codex | [理由] |
```

**完成检查：**
- [ ] 轮次 1 完成
- [ ] 轮次 2 完成（如需）
- [ ] 轮次 3 完成（如需）
- [ ] Plan 文档已定稿，状态改为"定稿"

## Phase 4：Codex 生成 Step-by-Step 开发计划

**工具：** Codex 插件版（gpt-5-codex）

**输入 Prompt：**

```
Plan 文档已定稿：[PLAN_DIR]YYYY-MM-DD-[FEATURE_NAME].md

请根据 Worktree 并行计划，为每个 worktree 生成独立的 step-by-step 开发计划：
- [STEPS_DIR]YYYY-MM-DD-[FEATURE_NAME]-task-1-steps.md
- [STEPS_DIR]YYYY-MM-DD-[FEATURE_NAME]-task-2-steps.md
- （按 worktree 数量生成对应数量的文件）

每个文件只包含该 worktree 负责的任务，格式要求（每个任务）：
### 任务 N：[任务名]

**文件：**
- 修改：`exact/path/to/file.js`
- 新建：`exact/path/to/new.js`
- 测试：`tests/exact/path/test.js`

**步骤 1：编写失败测试**
[完整测试代码，不是伪代码]

**步骤 2：运行测试确认失败**
命令：`npm test -- --grep "测试名"`
预期：FAIL，显示 "[具体错误信息]"

**步骤 3：实现最小代码**
[完整实现代码]

**步骤 4：运行测试确认通过**
命令：`npm test -- --grep "测试名"`
预期：PASS

**步骤 5：提交**
`git add [文件] && git commit -m "feat: [描述]"`

约束：
- 每个任务粒度：2-5 分钟可完成
- **Diff 上限：每个任务的代码改动 ≤ 200 行**（超过则继续拆分，大变更是 Review 9 小时的根因）
- 代码必须完整，不能用"添加验证"等模糊描述
- 测试必须描述行为，不描述实现细节
- 任务间依赖关系必须明确
```

**完成检查：**
- [ ] Step-by-step 开发计划已生成
- [ ] 每个任务有完整代码（非伪代码）
- [ ] 每个任务有明确验证命令

## Phase 4.5：任务解耦审查（三轮确认）

**为什么解耦是关键：** 解耦后的任务才能并行分配给多个 worktree + Codex 实例，互不干扰。解耦不充分会导致并行开发时产生冲突、覆盖彼此改动。

### 轮次 1：Claude Code 自查解耦

**工具：** Claude Code for VS Code，Plan Mode

```
请审查开发计划 [STEPS_DIR]xxx-steps.md，检查任务解耦质量：

对每个任务，验证：
1. 独立输入/输出：该任务的输入是否只依赖已完成的前置任务？
2. 文件边界清晰：该任务修改的文件列表是否与其他任务不重叠？
   （重叠 = 两个任务都修改同一个文件 → 必须拆分或明确顺序）
3. 验收标准独立：该任务的测试是否可以独立运行，不依赖其他任务的代码？
4. 无隐式依赖：是否有"假设任务X已经做了某件事"的隐式依赖？

输出：
- 解耦问题列表（文件冲突、隐式依赖、循环依赖）
- 建议的任务拆分或顺序调整
- 哪些任务可以并行执行（无依赖关系）
- 哪些任务必须串行执行（有依赖关系）
```

### 轮次 2：Codex 工程审查解耦

**工具：** Codex 插件版

```
请从工程实现角度审查任务解耦质量：[STEPS_DIR]xxx-steps.md

重点检查：
1. 文件修改冲突：列出所有被多个任务修改的文件
2. 函数/变量依赖：任务A新增的函数，任务B是否依赖？依赖关系是否已明确标注？
3. 测试隔离：各任务的测试是否互相独立，可以并行运行？
4. 并行安全性：如果同时在多个 worktree 中执行这些任务，会产生哪些 git 冲突？

输出解耦评分（1-5）和具体问题列表。
```

### 轮次 3：人工最终确认解耦

在开始开发前，人工过一遍以下清单：

```markdown
## 解耦确认清单

**文件边界检查：**
- [ ] 每个任务的"修改文件列表"互不重叠（或重叠已明确标注为串行）
- [ ] 没有两个任务同时修改同一个核心文件（如 index.js、config.js）

**依赖关系检查：**
- [ ] 任务依赖关系形成 DAG（有向无环图），无循环依赖
- [ ] 所有依赖关系已在计划文档中明确标注
- [ ] 并行任务组已确定（可同时在多个 worktree 执行）

**测试隔离检查：**
- [ ] 每个任务的测试可以独立运行
- [ ] 测试不依赖其他任务产生的数据或状态

**并行执行计划：**
| 批次 | 任务列表 | 分配 worktree |
|------|----------|--------------|
| 批次1（并行） | 任务1, 任务2, 任务3 | worktree-1, worktree-2, worktree-3 |
| 批次2（串行，依赖批次1） | 任务4, 任务5 | worktree-1, worktree-2 |
```

**完成检查：**
- [ ] Claude Code 自查解耦完成，问题已修复
- [ ] Codex 工程审查解耦完成，问题已修复
- [ ] 人工最终确认解耦通过
- [ ] 并行执行计划已确定（哪些任务可并行，哪些必须串行）
