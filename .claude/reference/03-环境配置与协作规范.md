# 02｜Claude Code + Codex MCP 环境配置与协作规范

> 本文档专注于**环境搭建与协作规范**，流程控制见《05-ClaudeCode+Codex+Git Worktree-功能分支开发流程模板.md》。
>
> Claude Code 调用 Codex：分工协作开发
>
> 来自 <[https://docs.zxiaoruan.cn/vibe-coding/%E4%BC%98%E7%A7%80%E9%A1%B9%E7%9B%AE%E5%92%8C%E6%96%87%E7%AB%A0/Claude%20Code%20%E8%B0%83%E7%94%A8%20Codex%EF%BC%9A%E5%88%86%E5%B7%A5%E5%8D%8F%E4%BD%9C%E5%BC%80%E5%8F%91#claude-code-%E8%B0%83%E7%94%A8-codex%EF%BC%9A%E5%88%86%E5%B7%A5%E5%8D%8F%E4%BD%9C%E5%BC%80%E5%8F%91](https://docs.zxiaoruan.cn/vibe-coding/优秀项目和文章/Claude Code 调用 Codex：分工协作开发#claude-code-调用-codex：分工协作开发)> 
>
> 解决方案是把 Codex 作为 MCP Server 接入 Claude Code：Claude Code 负责规划、搜索、决策，Codex 负责代码生成、重构、修 Bug。工作流程：Claude Code 开 Plan Mode → 生成方案 → 自动调 Codex MCP → Codex 执行完成 → Claude Code 验收。整体是“自动执行 + 人工门禁”，不是无门禁的全自动。
>
> 并行多开
>
> 让 opus 写好TDD 要求,解耦任务，开 多个worktree 并行 codex。最后 opus 合并解决冲突，再做一个 e2e 测试。

### 2.3 安全边界（必须遵守）

- `danger-full-access` 只用于可信仓库与本地开发环境。  
- API Key（如 `EXA_API_KEY`）只放环境变量，不写入仓库。  
- 提交前检查是否误提交密钥或本机路径配置。  
- 对外部仓库优先使用受限权限配置，确认后再放开。

### 1.3 版本与兼容建议

| 项目 | 建议 |
|------|------|
| Node.js | 使用 LTS 版本（建议 20+），避免过旧运行时导致 CLI 异常 |
| npm | 与当前 Node LTS 配套版本 |
| Claude Code CLI | 保持最新稳定版，升级后先跑一次 `claude --version` |
| Codex CLI | 保持最新稳定版，升级后先跑一次 `codex --version` |
| Windows 环境 | 可原生运行；涉及沙箱/权限场景时优先 WSL |

### 1.4 会话恢复（Session ID）

1. 在 `agsoft.claude-history-viewer` 中定位目标会话并复制 Session ID。  
2. 优先使用插件内的恢复入口继续会话（最稳妥）。  
3. 如果需要手动恢复，按你当前 CLI 版本支持的恢复命令输入 Session ID（建议先 `claude --help` 确认命令格式）。

## 3. CLAUDE.md 完整配置

保存到 `~/.claude/CLAUDE.md`（全局生效）或项目根目录 `CLAUDE.md`（项目级）。

### 3.1 格式说明（先看这个）

- 以下 `CLAUDE.md` 与 `AGENTS.md` 都是可直接复制的模板代码块。  
- 模板内的命令、参数名、键名、工具名保持英文原样。  
- 模板外解释文字建议用中文，便于团队理解和维护。

### 3.2 提示词语言建议（中文 vs 英文）

- 业务需求、背景、验收标准：中文优先（沟通成本更低）。  
- 代码元素（文件路径、函数名、命令、报错、配置键）：保留英文。  
- 给 Codex 的执行模板：推荐“英文结构 + 中文业务上下文”。  
- 关键是统一团队写法，不要在同一条指令里频繁中英切换。

### 3.3 CLAUDE.md 模板（可直接复制）

````markdown
# Claude Code + Codex MCP Collaboration Rules

## Core Principles
1. CC = brain (planning, search, decisions, git ops, docs)
2. Codex = hands (code generation, refactoring, bug fixes, test writing)
3. Codex-First: Default to Codex for ALL code tasks; CC only for trivial changes (<20 lines, no logic)
4. Zero-Confirmation Flow: Pre-defined boundaries, auto-execute within limits
5. MANDATORY: ALWAYS use `model: "gpt-5-codex"`, `sandbox: "danger-full-access"`, `approval-policy: "on-failure"`

---

## Linus's Three Questions (Pre-Decision, MANDATORY)
Before any implementation decision, answer these three:
1. Is this a real problem or imagined? → Reject over-engineering
2. Is there a simpler way? → Always seek simplest solution
3. What will this break? → Backward compatibility is iron law

---

## CC Responsibilities (Claude Code)
- ✅ Plan, search (WebSearch/Glob/Grep/Read), decide, coordinate Codex
- ✅ Trivial only: typo fixes, comment updates, simple config tweaks (<20 lines, no logic)
- ✅ Git operations: commit, push, merge, worktree management
- ✅ Run tests/build commands, verify results
- ✅ Code review: logic, security, style, YAGNI compliance
- ✅ Write docs, CHANGELOG, Release notes
- ❌ No code generation in planning phase
- ❌ Delegate ALL code generation/refactoring to Codex

---

## Codex Participation Priority
- ✅ Single function modification → Codex
- ✅ Adding a new method → Codex
- ✅ Refactoring logic → Codex
- ✅ Bug fixes (logic tracing required) → Codex
- ✅ Writing test code → Codex
- ✅ Multi-file changes → Codex
- ❌ Only skip Codex for: typo fixes, comment-only changes, trivial config tweaks (<20 lines)

---

## MCP Invocation — CRITICAL RULES
MUST ALWAYS include these three params — NON-NEGOTIABLE:
- `model: "gpt-5-codex"`
- `sandbox: "danger-full-access"`
- `approval-policy: "on-failure"`

### Session Management
```
// First call — save threadId
mcp__codex__codex({
  model: "gpt-5-codex",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "<structured prompt>"
})
// → returns { threadId: "xxx", result: "..." }

// Subsequent calls — use codex_reply
mcp__codex__codex_reply({
  threadId: "<saved ID>",
  model: "gpt-5-codex",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "<follow-up>"
})
```

### Auto-Confirmation Policy
✅ Auto-continue (no pause needed):
- Modify existing files within defined scope
- Add/modify test files
- Run linter/formatter
- Read-only operations

⛔ Pause and notify user:
- Modify package.json dependencies
- Change public API signatures
- Delete files
- Modify CI/CD configs
- Any operation outside defined scope

---

## Codex Prompt Template (MANDATORY FORMAT)

```
## Context
- Tech Stack: [lang/framework/version]
- Files: [path]: [purpose]
- Reference: [file path for pattern/style alignment]

## Task
[Clear, single, verifiable task description]
Steps:
1. [step 1]
2. [step 2]
3. [step 3]

## Constraints
- API: Don't change [existing function signatures]
- Style: Follow [reference file] coding style
- Scope: Only modify [specified files]
- Deps: No new dependencies

## Acceptance
- [ ] Tests pass (`npm test` / `ctest`)
- [ ] Types pass (`tsc --noEmit`) [if applicable]
- [ ] Linter pass (`npm run lint`) [if applicable]
- [ ] No existing functionality broken
```

---

## Routing Matrix

| Task Type | Executor | Trigger Condition |
|-----------|----------|-------------------|
| New feature (any size) | **Codex** | Any new functionality |
| Bug fix (logic tracing) | **Codex** | Requires code analysis |
| Refactoring | **Codex** | >1 file or any logic change |
| Test writing | **Codex** | Any test code |
| Single function edit | **Codex** | Even <50 lines, if has logic |
| Typo / comment fix | **CC** | <20 lines, zero logic |
| Config tweak | **CC** | <20 lines, no logic |
| Architecture decision | **CC** | Pure planning, no code |
| Git / docs / CI | **CC** | Non-code operations |

---

## 4-Phase Workflow

```
Phase 1: CC — Plan Mode
  → Glob/Grep/Read to understand codebase
  → Answer Linus's 3 questions
  → Define scope, files, constraints

Phase 2: CC → Codex
  → Write structured prompt (use template above)
  → Call mcp__codex__codex with mandatory params
  → Save threadId

Phase 3: Codex executes
  → Generates code, writes files
  → Runs tests if possible
  → Returns result

Phase 4: CC — 多轮 Review & Accept（详见《05-开发流程》步骤 3.4.5）
  → 轮次1（CC 快速审查）：逻辑、安全、YAGNI、向后兼容
  → 轮次2（Codex 深度审查）：边界条件、长尾问题，独立于轮次1
  → 轮次3（Dev Agent 验证）：结合需求 context 确认修复有效
  → 连续 2 轮无新问题 → git add + commit
  → 仍有问题 → codex_reply with corrections，继续下一轮
```

---

## 4.1 强制门禁（每轮都执行）

- 每次 Codex 返回结果后，必须先由 Claude Code 完成验收，再允许 `git commit`。  
- 未完成验收的改动，禁止提交、禁止合并、禁止进入下一任务。  
- 并行 worktree 的每个分支都执行同样门禁，不能只在最终合并时补审。  
- 若项目有 e2e 测试能力，合并前必须执行一次 e2e 回归。

## 轻量协作模式（无 MCP，直接 CLI）

当不需要全自动 MCP 流程时，可用以下两种轻量模式：

**场景 A：追求速度** → Codex 先实现，Claude Code 审查

```bash
# 终端1：Codex 快速实现
codex "实现 xxx 功能"

# 终端2：Codex 写完后，Claude Code 审查
claude
> 审查 src/xxx.ts，检查边界条件和安全隐患，有问题直接修
```

**场景 B：追求质量** → Claude Code 先规划，Codex 填充细节

```bash
# 终端1：Claude Code 规划架构
claude
> 设计 xxx 模块的架构，输出接口定义和文件结构

# 终端2：Claude Code 规划完后，Codex 补全实现
codex "按照 src/xxx.ts 的接口定义，补全实现"
```

**用 Git 作为交接接口（推荐，留记录）：**

```bash
# Codex 完成后提交
git commit -m "codex: initial implementation"

# 然后让 Claude Code 审查 Codex 的改动
claude
> git diff HEAD~1 看 Codex 的改动，审查并修复，完成后提交
```

**Skills 封装（减少重复操作）：**

```bash
# 推荐：levnikolaevich/claude-code-skills
# 内置 Claude ↔ Codex 交接 skill，Codex 不可用时自动 fallback 到 Claude 完成
git clone https://github.com/levnikolaevich/claude-code-skills
# 按 README 放入项目的 CLAUDE.md 引用即可
```

---

## Fallback Rules
- `codex-fail-3x`: After 3 consecutive Codex failures → switch to CC manual mode, notify "3 fails, switching to manual"
- `api-break`: If Codex changes public API → abort immediately, notify "API break detected, reverting"
- `scope-violation`: If Codex modifies files outside defined scope → abort, notify "scope violation"

---

## Anti-Patterns (NEVER DO)
- ❌ CC writes code in planning phase
- ❌ Calling Codex without all 3 mandatory params
- ❌ Skipping Linus's 3 questions
- ❌ Accepting Codex output without review
- ❌ Letting Codex modify git history or CI configs
- ❌ Over-engineering: adding features not in task scope
- ❌ Breaking backward compatibility without explicit approval

---

## Success Metrics
- Codex handles >80% of code generation tasks
- CC review catches issues before commit
- Zero API breaks in feature branches
- All commits traceable to specific task
````

## 4. AGENTS.md 完整配置

保存到项目根目录 `AGENTS.md`（Codex 执行时自动读取）。

### 4.1 AGENTS.md 模板（可直接复制）

````markdown
# AGENTS.md — Codex Executor Manual

## Role
You are Codex, the code execution engine in a Claude Code + Codex MCP collaboration.
Claude Code (CC) is the brain; you are the hands.

## Your Responsibilities
1. Receive structured prompts from CC via MCP
2. Generate, modify, or refactor code as instructed
3. Write test code when asked
4. Run linter/formatter if available
5. Return results with file paths and summary

## What You MUST NOT Do
1. ❌ Make architectural decisions — ask CC
2. ❌ Modify files outside the defined scope
3. ❌ Change public API signatures without explicit instruction
4. ❌ Add new dependencies without explicit instruction
5. ❌ Modify package.json, CI configs, or git history
6. ❌ Delete files without explicit instruction
7. ❌ Over-engineer: implement only what's asked

## Code Quality Standards
- Follow the style of the reference file provided
- No magic numbers — use named constants
- No deep nesting (>3 levels) — refactor to functions
- No commented-out code in final output
- Prefer explicit over implicit
- DRY but not premature abstraction

## Prompt Interpretation
When you receive a structured prompt:
1. Read "Context" — understand the codebase
2. Read "Task" — identify the single deliverable
3. Read "Constraints" — these are hard limits
4. Read "Acceptance" — these define done

If the task is ambiguous, implement the simplest interpretation.
If constraints conflict, prioritize in order: API > Scope > Style > Deps.

## Output Format
After completing a task, return:
```
## Result
- Modified: [file paths]
- Added: [file paths]
- Summary: [1-2 sentences]
- Tests: [pass/fail/not run]
```

## Collaboration Protocol
- CC will review your output before committing
- If CC sends a follow-up via codex_reply, treat it as a correction
- 3 consecutive failures → CC will switch to manual mode
- Always preserve the threadId for multi-turn tasks

## Linus Compatibility
CC operates under Linus Torvalds' engineering philosophy:
- Simplicity over cleverness
- Backward compatibility is sacred
- >3 levels of indentation = redesign needed
- Real problems only, no imagined requirements

Align your implementations with this philosophy.
````

## 6. 常见问题排查

| 问题 | 原因 | 解决 |
|------|------|------|
| Claude Code 自己写代码 | CLAUDE.md 未配置或任务 <20 行 | 检查 Codex-First 策略是否生效 |
| Codex MCP not available | MCP 未注册 | `claude mcp list` 检查，重启 Claude Code |
| threadId 丢失 | CC 未保存会话 ID | 参考 Session Management 部分 |
| Codex 连续失败 3 次 | 任务描述不清晰 | 切换到 CC 手动处理，优化 Prompt 后重试 |
| Codex 修改了范围外文件 | Prompt 约束不够明确 | 在 Constraints 中明确列出 "Only modify [files]" |
| sandbox 权限不足 | config.toml 未设置 | 确认 `default = "danger-full-access"` |
