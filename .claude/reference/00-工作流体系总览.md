# AI 开发工作流体系

> 基于 Claude Code + Codex MCP 的高质量软件开发规范
> 将 AI 首轮 60-70 分的代码质量，通过结构化流程提升到可交付标准

---

## 核心理念

```
不是让 AI 替代开发者，而是让 AI 做苦力，人做判断
规划 → 人决策    代码生成 → AI执行    质量把控 → 人+AI协作
```

**三条铁律：**
- 单次变更 ≤ 200 行（超了就拆，这是防止 Review 地狱的根本）
- AI 首轮代码必须多轮 Review（每轮只能召回部分问题）
- 计划先于代码（复杂任务不出 Plan 不动手）

## 全流程总览

```mermaid
flowchart TD
    A[收到任务] --> B{复杂度判断}
    B -->|diff≤200行\n≤3文件\n需求明确| C[简单模式]
    B -->|否| D[复杂模式]
    C --> C1[Plan Mode 分析\nLinus三问]
    C1 --> C2[Codex 实现]
    C2 --> C3[2轮独立Review]
    C3 -->|有问题| C2
    C3 -->|连续2轮无新问题| C4[commit]
    D --> D1[Opus 需求澄清\n生成Plan文档]
    D1 --> D2[Plan交叉Review\nOpus ↔ Codex ≥3轮]
    D2 --> D3[任务解耦\n每任务≤200行]
    D3 --> D4{人工复核\n解耦是否通过}
    D4 -->|不通过| D3
    D4 -->|通过| D5[并行Worktree开发]
    D5 --> D6[多轮Review\n每任务独立]
    D6 --> D7[Opus主导合并\ne2e测试门禁]
    D7 --> D8[推送main]
    C4 --> E[完成]
    D8 --> E
```

## 简单模式流程

```mermaid
sequenceDiagram
    participant H as 人类
    participant CC as Claude Code
    participant CX as Codex MCP
    H->>CC: 描述任务
    CC->>CC: Plan Mode：Linus三问\n列出改动文件
    CC->>H: 确认改动范围
    H->>CC: 确认
    CC->>CX: 调用MCP执行代码生成
    CX-->>CC: 返回代码 + threadId
    CC->>CC: 轮次1：自身审查\n逻辑/安全/YAGNI
    CC->>CX: 轮次2：独立深度审查
    CX-->>CC: 边界条件/长尾问题
    CC->>H: Review结果汇总
    H->>CC: 确认无问题
    CC->>CC: git commit
```

## 快速开始

```bash
# 1. 把工作流规范放入项目
cp CLAUDE.workflow.md /your-project/

# 2. 启动 Claude Code，它会自动读取规范
cd /your-project
claude

# 3. 开始第一个任务（简单模式示例）
# 在 Claude Code 中输入：
# "分析 [你的任务]，先不要改代码，回答Linus三问后等我确认"
```
