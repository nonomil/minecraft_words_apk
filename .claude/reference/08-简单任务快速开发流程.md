# 07｜简单任务快速开发流程（Simple Mode）

> 适用条件：diff ≤ 200 行、≤3 文件、需求明确、你熟悉代码。
> Token 消耗约为复杂模式的 1/5，适合日常迭代。
> 不确定用哪种模式？先做下方的复杂度判断。

---

## Phase -1：复杂度判断 → 选择开发模式

**在任何开发开始前，先做这个判断。**

```
1. 涉及文件数量？
   ≤ 3 个文件 → 简单
   > 3 个文件 → 复杂倾向

2. 预估 diff 行数？
   ≤ 200 行 → 简单
   200-500 行 → 中等
   > 500 行 → 复杂（必须拆分）

3. 是否跨模块？
   单模块内 → 简单
   跨模块 → 复杂倾向

4. 需求是否明确？
   需求清晰，边界明确 → 简单
   有歧义，需要澄清 → 复杂倾向

5. 你对这块代码的熟悉程度？
   熟悉 → 简单倾向
   不熟悉 → 复杂倾向

6. 任务能否拆成 3+ 个独立单元并行？
   不能 → 简单模式（并行优势不存在）
   能 → 复杂模式有价值
```

### 模式选择

```
简单模式（≥4个"简单"答案）：
  → 跳过复杂流程
  → 直接用 Claude Code for VS Code + Plan Mode 分析
  → 单分支开发，CC 审查 2 轮
  → 按本文档执行

复杂模式（≥3个"复杂倾向"答案）：
  → 执行完整 Plan Phase（见《04-AI计划阶段》）
  → 多 worktree 并行开发（见《05-开发流程模板》）

中间地带（难以判断）：
  → 先用简单模式启动
  → 触发升级条件时切换到复杂模式（见下方）
```

## 简单开发流程（4 步）

### Step 1：Plan Mode 分析

打开 Claude Code for VS Code，按 `Shift+Tab` 进入 Plan Mode，输入：

```
分析 [具体任务]，在修改任何代码前：
1. 搜索相关文件（Glob/Grep）
2. 回答 Linus 三问：
   - 这是真实问题还是想象的？
   - 有没有更简单的方式？
   - 这会破坏什么现有功能？
3. 列出将要修改的文件和改动点
4. 等我确认后再执行
```

确认无误后退出 Plan Mode。

- [ ] Plan Mode 分析完成，改动范围已确认

---

### Step 2：实现

- 仅限微小改动（<20 行且无逻辑）→ Claude Code 直接实现
- 有逻辑的改动 → Codex MCP 实现（保留 threadId 供后续追问）

- [ ] 实现完成

---

### Step 3：快速 Review（2 轮）

**轮次 1：CC 审查（逻辑、安全、YAGNI）**

```
[Claude Code，正常模式]
请审查刚才对 [文件列表] 的修改，任务是：[任务描述]
重点：逻辑正确性、安全性（XSS/注入）、YAGNI、向后兼容。
输出问题列表，无问题则说"通过"。
```

**轮次 2：Codex 深度审查（边界条件、长尾）**

```
mcp__codex__codex({
  model: "gpt-5-codex",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "请审查 [文件列表] 的修改，任务是：[任务描述]
           重点：边界条件、长尾情况、与现有代码的兼容性。
           输出问题列表，无问题则说'通过'。"
})
```

连续 2 轮无新问题 → 进入 Step 4。
仍有问题 → 修复后重新从轮次 1 开始（最多 3 轮，超过则触发升级）。

- [ ] 2 轮 Review 通过

---

### Step 4：提交

```bash
git diff --stat   # 确认 ≤ 200 行
git add [具体文件]
git commit -m "feat/fix: [描述]"
```

- [ ] diff ≤ 200 行，已提交

---

## 模式升级触发条件（简单 → 复杂）

开发过程中，出现以下任一情况，**立即暂停，切换到复杂模式**：

```
触发条件：
□ diff 超过 200 行且还没完成
□ 发现需要修改 >3 个文件
□ 出现跨模块依赖，原本以为是单模块
□ 需求在开发中变化，边界扩大
□ Review 超过 3 轮仍有新问题（说明任务太大）
□ 发现任务可以拆成 3+ 个独立单元

切换步骤：
1. 停止当前开发，先执行 `git status --short` 并汇报当前改动
2. 让用户选择进度处理方式：A 提交当前改动 / B 暂存（仅用户明确同意后）/ C 放弃（高风险，需明确确认）
3. 打开《04-AI计划阶段》Phase 1，重新做需求澄清
4. 把已完成的部分作为"已知背景"告知 Opus
5. 重新生成 Plan 和 step-by-step 计划
6. 按《05-开发流程模板》复杂模式继续
```
