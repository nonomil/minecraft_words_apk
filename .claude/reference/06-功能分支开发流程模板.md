# 05｜Claude Code + Codex + Git Worktree 功能分支开发流程模板

> 通用模板：适用于任何基于 Git + CI/CD 的项目。
> 支持：远端 CI APK 构建 / 本地 CMake exe 编译 / Claude Code + Codex MCP 全自动化协作。
> 将 `[PROJECT_*]` 占位符替换为实际项目信息后使用。
>
> **环境配置与协作规范**（安装、CLAUDE.md、AGENTS.md、MCP JSON）见：
> `docs/development/02-ClaudeCode+Codex-环境配置与协作规范.md`

---

## Claude Code + Codex MCP 协作架构

```
┌─────────────────────────────────────────────────────┐
│                   Claude Code  (大脑)                │
│  规划 / 搜索 / 决策 / 验收 / 文档 / Git 操作         │
│  Plan Mode → 生成方案 → 调用 Codex MCP → 验收结果   │
└──────────────────┬──────────────────────────────────┘
                   │ MCP Tool Call (自动)
                   ▼
┌─────────────────────────────────────────────────────┐
│                 Codex MCP Server  (双手)             │
│  代码生成 / 重构 / Bug 修复 / 测试代码生成            │
│  接收结构化 Prompt → 执行代码任务 → 返回结果         │
└─────────────────────────────────────────────────────┘
```

**分工速查：**

| 职责 | Claude Code | Codex MCP |
|------|-------------|-----------|
| 规划/搜索/架构决策 | ✅ | ❌ |
| 代码生成/重构/Bug修复 | ❌ | ✅ |
| 测试代码生成 | ❌ | ✅ |
| 拼写/注释/简单配置（<20行） | ✅ | ❌ |
| 运行测试/构建/Git操作 | ✅ | ❌ |

```
Phase 9   推送 main 到远端 + 等待 CI
Phase 10  完成总结报告 + 计划文档打勾
```

## Phase 3：TDD 逐任务实现（Claude Code + Codex MCP 协作）

对计划文档中的每个任务，重复以下循环：

**步骤 3.1：Claude Code — 开 Plan Mode，搜索现有代码**

```
Shift + Tab  →  进入 Plan Mode
```

使用 Grep/Glob/Read 搜索相关文件，回答 Linus 三问：
1. 这是真实问题还是想象的问题？
2. 有没有更简单的方式？
3. 这会破坏什么现有功能？

**步骤 3.2：Claude Code — 写失败测试**

新建或修改测试文件，测试描述期望行为而非实现细节。

**步骤 3.3：运行测试，确认失败**

*npm 项目：*
```bash
npm test -- --grep "测试名称"
# 预期：FAIL
```

*CMake 项目：*
```bash
cd build && ctest -R "测试名称" -V
# 预期：FAIL
```

**步骤 3.4：Claude Code 调用 Codex MCP 生成实现代码**

使用结构化 Prompt，必须包含三个强制参数：

```
mcp__codex__codex({
  model: "gpt-5-codex",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "
    ## Context
    - Tech Stack: [语言/框架/版本]
    - Files: [文件路径]: [文件用途]
    - Reference: [参考文件路径，用于风格对齐]

    ## Task
    [清晰、单一、可验证的任务描述]
    Steps:
    1. [步骤1]
    2. [步骤2]

    ## Constraints
    - API: Don't change [已有函数签名]
    - Scope: Only modify [指定文件]
    - Deps: No new dependencies
    - Style: Follow [参考文件] coding style

    ## Acceptance
    - [ ] Tests pass (npm test)
    - [ ] No existing functionality broken
  "
})
// 保存返回的 threadId，后续追问用 codex_reply
```

**后续追问（同一会话）：**
```
mcp__codex__codex_reply({
  threadId: "<上一步返回的 ID>",
  model: "gpt-5-codex",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "请补充错误处理：[具体要求]"
})
```

**Claude Code 验收 Codex 输出：**
- 逻辑是否正确
- 是否有安全问题（XSS、注入等）
- 是否与现有代码风格一致
- 是否超出最小实现范围（YAGNI）
- 向后兼容性是否有破坏

门禁规则：未通过以上验收，不得进入提交步骤。

> Codex 连续失败 3 次 → 切换到 Claude Code 手动处理，优化 Prompt 后重试

**步骤 3.4.5：多轮 Review 协议（每次 Codex 执行后必须执行）**

> 依据实践：AI 首轮代码质量约 60-70 分，每轮 Review 只能召回部分问题（需"抽卡"）。
> 必须至少 3 轮 Review（轮次1 CC 快速审查、轮次2 Codex 深度审查各自独立进行，轮次3 Dev Agent 结合需求 context 验证），直到连续 2 轮无新问题才可提交。

**Diff 上限检查（Review 前先检查）：**
```bash
git diff --stat   # 单次任务 diff 应 ≤ 200 行
# 超过 200 行 → 拆分任务，不要继续 Review，先回到《04-AI计划阶段》Phase 4.5 重新解耦
```

**止损阈值（触发后强制拆分任务，不得继续 Review）：**
- 单任务 Review 超过 5 轮仍有新问题
- 单任务 Review 累计超过 3 小时
- 单任务 Token 消耗超过 50k

触发后：停止当前 Review → 先执行 `git status --short` 汇报改动 →
让用户选择提交/暂存（仅用户明确同意）/放弃（高风险） →
再回到《04-AI计划阶段》Phase 4.5 重新解耦该任务。

**轮次 1：Claude Code 快速审查（速度优先）**
```
[Claude Code，正常模式]
请审查 Codex 刚才对 [文件列表] 的修改，任务是：[任务描述]

重点检查：
1. 逻辑正确性：实现是否符合任务？有无遗漏的边界条件？
2. 安全性：XSS、注入、敏感数据泄露？
3. YAGNI：有无超出任务范围的额外实现？
4. 向后兼容：是否改变了现有公共 API 或数据格式？

输出：问题列表（按严重程度排序），无问题则说"通过"。
```

**轮次 2：Codex 深度审查（质量优先，独立于轮次1）**
```
mcp__codex__codex({
  model: "gpt-5-codex",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "
    ## Review Task（独立审查，不参考轮次1结论）
    请深度审查以下文件的修改：[文件列表]
    任务背景：[任务描述]

    审查维度：
    1. 长尾边界条件（空值、最大值、并发、异常路径）
    2. 代码完善性（死代码、冗余逻辑、未处理的错误）
    3. 测试覆盖（测试是否覆盖了主路径和边界条件）
    4. 与现有代码的一致性（风格、命名、模式）

    输出：候选问题列表，标注每个问题的严重程度（高/中/低）。
  "
})
```

**Dev Agent 验证（Claude Code 结合需求 context 筛选）：**
```
[Claude Code]
Codex Review 发现了以下候选问题：[Codex 问题列表]
Claude Code 轮次1 发现了：[CC 问题列表]

请结合任务需求和开发历史，逐条判断：
1. 这个问题是真实问题还是误报？
2. 如果是真实问题，严重程度如何？
3. 是否需要立即修复，还是可以记录为 tech debt？

输出：确认需要修复的问题列表。
```

**修复（Codex MCP）：**
```
mcp__codex__codex_reply({
  threadId: "<Review 的 threadId>",
  model: "gpt-5-codex",
  sandbox: "danger-full-access",
  "approval-policy": "on-failure",
  prompt: "请修复以下确认问题：\n[Dev Agent 确认的问题列表]"
})
```

**轮次 3：验证修复 + 终止判断**
```
[Claude Code]
请验证上轮修复是否正确，并检查修复是否引入了新问题。
如果无新问题 → 通过，进入步骤 3.5。
如果有新问题 → 继续下一轮（重复轮次2-3）。
```

**终止条件：** 连续 2 轮无新问题 → 通过。
**上限：** 超过 5 轮仍有问题 → 任务粒度太大，拆分后重做。

**Context 刷新策略（防止降智）：**
- 单个 Codex Session 执行超过 3 个任务 → 开新 Session，重新提供 context
- 发现明显低质量输出（逻辑混乱、忽略约束）→ 立即重启 Session
- 北京时间 21:00 后如发现质量下降 → 重启 Session 或改用 Claude Code 手动处理

**步骤 3.5：写入文件，运行测试确认通过**

*npm：* `npm test -- --grep "测试名称"`
*CMake：* `cmake --build build && ctest -R "测试名称" -V`

**步骤 3.6：运行全量测试，确认无回归**

*npm：* `npm test`
*CMake：* `cmake --build build && ctest --output-on-failure`

**步骤 3.7：提交**
```bash
git diff --stat   # 再次确认 diff ≤ 200 行
git add [修改的文件列表]
git commit -m "feat: [任务描述]"
```


- [ ] 任务 1 完成并提交
- [ ] 任务 2 完成并提交
- [ ] 任务 3 完成并提交
- [ ] （根据计划文档任务数量增减）
- [ ] 每个任务提交前均已完成 Claude Code 验收

### 5.2 创建 push-branch.bat

在 `[DEV_DIR]` 下创建 `push-branch.bat`：

```batch
@echo off
chcp 65001 >nul 2>&1
set "REMOTE=origin"

for /f "delims=" %%A in ('git rev-parse --abbrev-ref HEAD 2^>nul') do set "CURRENT_BRANCH=%%A"
if not defined CURRENT_BRANCH ( echo [错误] 无法获取分支名 & pause & exit /b 1 )
if /i "%CURRENT_BRANCH%"=="main" ( echo [警告] 在 main 分支，请用 push.bat & pause & exit /b 1 )

echo [信息] 推送分支: %CURRENT_BRANCH%

set "PROXY_OK=0"
for /f "delims=" %%P in ('powershell -NoProfile -Command "$ok=0;try{$c=New-Object System.Net.Sockets.TcpClient;$iar=$c.BeginConnect('127.0.0.1',1080,$null,$null);if($iar.AsyncWaitHandle.WaitOne(200,$false)-and $c.Connected){$ok=1};$c.Close()}catch{};Write-Output $ok" 2^>nul') do set "PROXY_OK=%%P"

if "%PROXY_OK%"=="1" (
    git -c http.proxy=http://127.0.0.1:1080 -c https.proxy=http://127.0.0.1:1080 -c http.sslBackend=openssl push %REMOTE% %CURRENT_BRANCH%
) else (
    git -c http.version=HTTP/1.1 push %REMOTE% %CURRENT_BRANCH%
)

if errorlevel 1 ( echo [失败] 推送失败 & pause & exit /b 1 )
echo [成功] %CURRENT_BRANCH% 已推送到 %REMOTE%
pause
```

```bash
git add push-branch.bat
git commit -m "chore: add push-branch.bat"
```

## AI 执行安全约束（给 Claude Code 的强制规则）

### 根因：哪些操作会导致主分支改动丢失

1. **`git stash -u` 隐式收走未跟踪文件**：主分支本地未提交的新文件（如文档）会被藏进 stash，看起来像"被删了"，实际在 stash 里。
2. **提前合并/推送 main**：未等 CI 绿灯就执行 merge/push，破坏"分支推送 → CI通过 → 再合main"的可控节奏。
3. **worktree 合并时未先 `pull --ff-only`**：主分支落后远端时强行合并，可能覆盖远端已有改动。

### 强制操作约束

以下约束适用于所有 AI 执行者（Claude Code / Codex / 子代理），**未收到用户明确批准前禁止执行**：

```text
严格流程约束：
1) 未经用户明确批准，禁止执行：
   git stash / git stash -u / merge main / push main / reset --hard / checkout -- . / clean -f
2) 只允许在 worktree 功能分支开发与提交，不得在主仓库 main 分支直接改动。
3) 功能分支推送后必须等待用户确认"CI通过"，才能合入 main。
4) 每次切分支/创建 worktree 前先汇报 git status；若非干净，停下让用户选处理方式。
5) 合并前先给出将变更的文件清单和 commit 列表，等用户回复"确认合并"再执行。
6) docs/development/** 未经用户点名不得修改、移动、删除。
```

### Phase 1 的正确处理流程

```
AI 执行 git status --short
  ↓
有未提交改动？
  ├── 是 → 停下，向用户汇报改动列表，询问：
  │         A) 提交这些改动（git add -A && git commit）
  │         B) 暂存（git stash -u）—— 需用户明确选择
  │         C) 放弃改动（危险，需用户明确确认）
  │         等待用户回复后再继续
  └── 否 → 继续创建 worktree
```
