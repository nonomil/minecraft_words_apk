# 历史教训 (Lessons Learned)

> 本文件记录开发过程中的错误 pattern 和防范方式，供 CC+Codex 协作时参考。

## 格式规范
```
- [日期] [场景] → 错误根因 → 下次防范方式
```

## 教训记录

- [2026-03-04] 大型代码库扫描 → 本地 extract 零成本提取代码结构效率高 → 优先使用 extract 而非直接 Codex 扫描
- [2026-03-04] 大型代码库扫描 → 单模块统计（extract）与逻辑划分（21个模块）存在差异 → 需人工判断实际模块数，不能只看物理文件统计
- [2026-03-04] 大型代码库扫描 → 纯 JavaScript 项目缺少类型信息，函数签名需推测 → TypeScript 项目扫描质量更高
- [2026-03-04] 大型代码库扫描 → 策略 B（CC 直接执行）可作为 Codex 不可用时的兜底方案 → 基于 extract-data.json 生成结构化文档
- [2026-03-04] 并行任务规划 → 基于扫描结果的文件改动矩阵可准确判断任务是否可并行 → 创建文件改动矩阵避免并行冲突
- [2026-03-04] 并行任务规划 → 每个任务独立文档便于独立更新进度 → 避免多任务共享单一文档导致冲突
- [2026-03-04] 并行开发合并 → Worktree D 合并时引入语法错误（12-challenges.js:949 不完整的 catch 语句）→ 合并后必须立即运行语法检查（node -c）和冒烟测试
- [2026-03-04] 并行开发测试 → 初次测试时只运行了本次合并相关的测试（opt-0226-*），未运行完整测试套件 → 合并后应运行完整测试套件（包括 opt-0226-* 全部 10 个测试）发现所有问题
- [2026-03-04] 并行开发测试 → task3 测试失败（按钮尺寸 57.6px < 60px）但初次测试时未发现 → 因为只选择性运行了 task1/5/6/7，跳过了 task2/3/4
- [2026-03-04] 测试策略 → 选择性测试会遗漏问题，应该运行完整测试套件 → 合并后必须运行所有相关测试（opt-0226-*.spec.mjs 全部 10 个），不能只挑选部分
- [2026-03-04] 代码审查 → 合并时未仔细审查 Worktree D 的代码改动，导致语法错误进入主分支 → 合并前必须对每个 worktree 的代码进行语法检查和基本验证
- [2026-03-04] 文档编码 → Windows GBK 终端链路 + `errors='replace'` 会把中文静默替换为 `?`/`�` 且不可逆 → 中文文档统一 UTF-8（建议 BOM），命令前设置 `PYTHONIOENCODING=utf-8`，禁止 replace 后写回
